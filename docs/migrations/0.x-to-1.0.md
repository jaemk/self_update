# Migrating to `self_update` 1.0 (automation guide)

Scope: upgrading a downstream crate's usage of `self_update` from `0.x` to `1.0.0`.
Apply every rule below in order. Rules are either **mechanical** (deterministic
find/replace) or **manual** (needs local context — flag for review, do not blind-apply).
After applying, run the **Verify** step and resolve any remaining compiler errors using the
"Compiler-error → fix" table.

---

## 0. Dependency

- File: `Cargo.toml`
- Mechanical: set the `self_update` dependency version requirement to `"1.0"` (or `"1"`).

## 1. Builder setter renames (mechanical, method-call level)

Apply as literal substring replacements on Rust source (`.rs`) files. Each pattern includes
the leading `.` and trailing `(` so only method calls match (never field/type names). These
patterns match on method name only, not receiver type: if your code calls a same-named method
(e.g. `.timeout(`, `.header(`, `.show_progress(`) on a value that is **not** a `self_update`
builder or `Download`, scope the replacement to the `self_update` call sites named in each row's notes.

| # | find | replace | notes |
|---|------|---------|-------|
| 1.1 | `.with_url(` | `.url(` | github `Update`/`ReleaseList` builders (the github API-endpoint setter; **kept** as `url`) |
| 1.2 | `.with_host(` | `.instance_url(` | gitlab + gitea builders (the custom-endpoint setter — **B2**). gitea has **no default host**, so `.instance_url(...)` is required — `build()` errors with `` `instance_url` required `` if missing (gitlab still defaults to `https://gitlab.com`). Also covers a `.url(` from an earlier 1.0 pre-release on gitlab/gitea. |
| 1.3 | `.with_target(` | `.filter_target(` | the `ReleaseList` release-filter setter on all four backends (**B3**). NOTE: the `Update` builder's own `.target(` (build-target triple, from the shared macro) is **unchanged** — do not rewrite it. Also covers a `.target(` on a `ReleaseList` builder from an earlier 1.0 pre-release. |
| 1.4 | `.access_key_id(` | `.access_key(` | s3 `Update` builder (`s3-auth`); the s3 `ReleaseList` builder already used `.access_key(` and is unchanged |
| 1.5 | `.set_headers(` | `.replace_headers(` | `Download` — replaces the whole `HeaderMap` (the additive header setter is now `header`, see 1.10) |
| 1.6 | `.show_progress(` | `.show_download_progress(` | `Download` — renamed to match the `Update` builder setter of the same name |
| 1.7 | `.target_version(` / `.target_version_tag(` | `.release_tag(` | `.target_version(` was the 0.x **accessor** call site; `.target_version_tag(` was the 0.x **setter** — both collapse to `.release_tag(` (the `Update` builder setter and the `ReleaseUpdate` accessor now share the name, **B5**; also see §3 for the accessor's return-type change). |
| 1.8 | `.identifier(` | `.asset_identifier(` | the asset-disambiguation setter on every `Update` builder and the `ReleaseUpdate` accessor (**B5**); set and get share the name `asset_identifier`. |
| 1.9 | `.set_progress_callback(` | `.progress_callback(` | `Download` and every `Update` builder (**B1**); the `Update` accessor is also `progress_callback` (set/get match — **B4**). |
| 1.10 | `.set_progress_style(` | `.progress_style(` | `Download` and every `Update` builder (**B1**). |
| 1.11 | `.set_timeout(` | `.timeout(` | `Download` (**B1**) — matches the `Update`/`ReleaseList` builders' `.timeout(`. |
| 1.12 | `.set_header(` | `.header(` | `Download` additive single-header setter (**B1**). Distinct from `replace_headers` (1.5). |
| 1.13 | `.verifying_checksum(` | `.checksum(` | every `Update` builder (`checksums` feature) (**B4**); matches the `checksum()` accessor (set/get align). |

These renames are unambiguous: the old names no longer exist in 1.0, so any remaining
occurrence is a real call site to convert. The old method names are also kept as `#[doc(alias)]`s,
so searching the rustdoc for an old name points at the new one.

## 1a. `request_header` now takes `TryInto` args (mechanical, non-breaking)

The `Update`/`ReleaseList` builder setter `.request_header(name, value)` now accepts anything
convertible to a header name/value, so a previously required `.parse().unwrap()` can be dropped:
`.request_header("X-Foo", "bar")` and
`.request_header(self_update::http::header::ACCEPT, "application/json")` both compile. An invalid
header name/value is no longer a panic at the call site — it is surfaced as an `Error::Config`
from `build()`. (Existing typed-argument call sites still compile via the identity conversion.)

`Download::header` (the additive single-header setter on the standalone downloader, renamed from
`set_header` — see row 1.12) still takes concrete `HeaderName`/`HeaderValue` arguments and is
**unchanged** — callers still need `.parse()` for string values. It is doc-aliased as `set_header`
so rustdoc search finds it under the old name.

## 2. Error enum (manual — semantic)

`self_update::errors::Error` is now `#[non_exhaustive]` and feature-specific variants were
collapsed. For every `match` (or `if let`) on an `Error` value:

- Replace arm patterns:
  - `Error::Reqwest(e)` → `Error::Http(e)`
  - `Error::Ureq(e)` → `Error::Http(e)`
  - `Error::StdTimeError(e)` → `Error::S3Auth(e)`
  - `Error::TimeError(e)` → `Error::S3Auth(e)`
  - `Error::Digest(e)` → `Error::S3Auth(e)`
  - `Error::UrlParse(e)` → `Error::S3Auth(e)`
- In `Error::Http(e)` / `Error::S3Auth(e)`, `e` is now `Box<dyn std::error::Error + Send + Sync>`
  (was a concrete `reqwest::Error` / `ureq::Error` / etc.). If code called a concrete method on
  the inner error, downcast: `e.downcast_ref::<reqwest::Error>()`, or read it via
  `std::error::Error::source()` on the outer `Error`.
- Because `Error` is now `#[non_exhaustive]`, every external `match` on it MUST have a wildcard
  arm. If a `match` is now non-exhaustive, add `_ => { /* handle/ignore */ }`.
- Feature gating: `Error::S3Auth` only exists when the `s3-auth` feature is enabled (and
  `Error::Zip` / signature variants under their features); `Error::Http` is always present. If
  the consumer does not enable `s3-auth`, omit the `S3Auth` arm — the required `_ =>` arm
  already covers it. Do NOT add an `Error::S3Auth` arm to a crate that doesn't enable `s3-auth`
  (it won't compile).

If two of the old arms collapse to the same new variant in one `match` (e.g. both `Reqwest`
and a later `Ureq` arm → `Http`), keep ONE `Error::Http(e)` arm and delete the duplicate.

**Also `#[non_exhaustive]` in 1.0:** `Status`, `ArchiveKind`, and `Compression` are now also
`#[non_exhaustive]`. An exhaustive `match` on any of them — especially `Status`, returned by
every `update()` call — must gain a `_ =>` arm. The compiler will flag any non-exhaustive pattern.

## 3. `ReleaseUpdate` accessor return types (manual — type)

The `ReleaseUpdate` trait accessors now return borrows:
- `current_version`, `target`, `bin_name`, `bin_path_in_archive`, `progress_template`,
  `progress_chars` return `&str` (were `String`).
- `release_tag` (renamed from `target_version`/`target_version_tag` — see §1.7),
  `asset_identifier` (renamed from `identifier` — see §1.8), `auth_token`
  return `Option<&str>` (were `Option<String>`).
- `verifying_keys` (`signatures` feature) returns `&[VerifyingKey]` (was `Vec<VerifyingKey>`).

Fix sites where ownership is required: append `.to_string()` / `.map(str::to_owned)`.
- `let v: String = updater.current_version();` → `let v: String = updater.current_version().to_string();`
- `let id: Option<String> = updater.identifier();` → `... = updater.asset_identifier().map(str::to_owned);`
Sites that only read (`==`, `format!`, passing `&str`) need no change.

## 4. `ReleaseUpdate` is sealed (manual — rare)

`impl self_update::update::ReleaseUpdate for MyType { .. }` no longer compiles (the trait is
sealed). There is no mechanical fix: a custom backend must be reworked to use a built-in
backend, or the upgrade blocked pending an upstream feature request. Flag for human review.

## 5. Feature flags (manual — `Cargo.toml`)

Invalid HTTP-client / TLS feature combinations are now a hard `compile_error!`:
- `reqwest` + `ureq` together → invalid. Pick one. For `ureq`, set
  `default-features = false` and enable `ureq` (+ one TLS feature).
- *neither* `reqwest` *nor* `ureq` → invalid (`no HTTP client selected`). This fires when you
  set `default-features = false` (e.g. to switch to `ureq` or `rustls`) but forget to enable a
  client; enable exactly one of `reqwest` or `ureq`.
- `default-tls` + `rustls` together → invalid. For `rustls`, set `default-features = false`.

If the manifest enables `self_update` with `features = ["ureq"]` or `["rustls"]` WITHOUT
`default-features = false`, add `default-features = false` and ensure exactly one client
(`reqwest` xor `ureq`) and one TLS backend (`default-tls` xor `rustls`).

**s3 alias feature (D1):** the S3 backend needs **no feature** (it is always compiled). A no-op
`s3` alias feature now exists so `features = ["s3"]` resolves for symmetry with the other
backends; it pulls in nothing. Private-bucket request signing still requires the real `s3-auth`
feature.

## 6. Removed / privatised items (manual — rare)

- **Dropped re-exports (C1):** `self_update::self_replace` and `self_update::TempDir` are no
  longer re-exported. If you referenced `self_update::self_replace::…` or `self_update::TempDir`,
  add the `tempfile` and/or `self-replace` crates as your own dependencies and import them
  directly (`tempfile::TempDir`, `self_replace::self_replace`). The `http`, `reqwest`/`ureq`, and
  `zipsign_api` re-exports are **unchanged**.
- **s3 `AccessKey` is now public and `#[non_exhaustive]` (C3):** the credential type is
  re-exported as `self_update::backends::s3::AccessKey` (under `s3-auth`). Build one through its
  `From<(&str,&str)>` / `From<(String,String)>` impls (i.e. pass an `(id, secret)` tuple to
  `.access_key(...)`), not a struct literal — `#[non_exhaustive]` leaves room for an additive
  field (e.g. an STS session token) later. No change needed unless you named the type.
- `self_update::GetArchiveReaderResult` is no longer public. It had no consumer use; delete
  any reference.
- `std::error::Error::description` is no longer implemented on `Error` (it was deprecated);
  use `Display` / `source()`.
- `self_update::should_update(current, latest)` is removed (deprecated since 0.4.2). Replace
  with `self_update::version::bump_is_greater(current, latest)` (same "is latest newer"
  semantics), or `version::bump_is_compatible(current, latest)` for a compatibility check.
- **Behavior change (Linux, `default-tls`):** the crate no longer sets `SSL_CERT_FILE` /
  `SSL_CERT_DIR` for you before each request (it used `std::env::set_var`, unsound in a
  multi-threaded process and hardcoded to Debian/Ubuntu paths). No code change is needed, but if
  you relied on that auto-setting in a minimal environment (some containers, `musl` static builds)
  and now hit a certificate-verification error, export those variables yourself, or build with the
  `rustls` feature. See the crate "Troubleshooting" docs.

## 7. Optional cleanups (non-breaking, safe to skip)

- The example header type `reqwest::header::ACCEPT` can become `self_update::http::header::ACCEPT`
  (the `http` crate is re-exported as `self_update::http`; no separate `http` dep needed).
- `[u8; zipsign_api::PUBLIC_KEY_LENGTH]` can be written `self_update::VerifyingKey` (signatures
  feature); the `zipsign_api` crate is re-exported as `self_update::zipsign_api`.
- `self_update::ReleaseUpdate` is now re-exported at the crate root (was
  `self_update::update::ReleaseUpdate`).
- s3 `.auth_token(` is now `#[deprecated]` (a compiler **warning**, not an error) and a no-op:
  the s3 backend authenticates via `access_key` (AWS SigV4), never a bearer token, so the call
  never did anything. Delete the `.auth_token(..)` call on s3 `Update` builders; if private-bucket
  auth was intended, replace it with `.access_key((id, secret))` (the `s3-auth` feature). This
  applies only to the **s3** backend — `auth_token` remains functional on github/gitlab/gitea.

## 8. Custom-backend async + accessor split (manual — rare)

Two changes to the custom-backend / `ReleaseUpdate` surface (all under the `async` feature except
the accessor split):

**8a. `ReleaseSource` is sync-only again.** The defaulted async helper methods
(`get_latest_release_async`, `get_latest_releases_async`, `get_release_version_async`) were removed
from [`ReleaseSource`]. The trait is now three sync fetches plus `Send + Sync`, with no
`Self: Clone` bound. If you implemented or called those `*_async` defaults:

- For a **natively-async** source, implement the new
  [`AsyncReleaseSource`](https://docs.rs/self_update/latest/self_update/trait.AsyncReleaseSource.html)
  trait instead — the same three fetches as `async fn` with **clean names** (no `_async` suffix:
  `get_latest_release`, `get_latest_releases`, `get_release_version`) — and drive the update with
  `self_update::backends::custom::AsyncUpdate::configure().source(..).build_async()?.update_async().await?`
  (mirrors a built-in backend's `build_async()`).
- To reuse an existing **`Clone` sync `ReleaseSource`** from the async API, wrap it:
  `AsyncUpdate::configure().source(self_update::backends::custom::Blocking::new(my_sync_source))`.
  `Blocking` runs the sync fetches on `tokio::task::spawn_blocking` (the same behavior the removed
  defaults provided).

`AsyncReleaseSource` is consumed through generics (`AsyncUpdate<S>`), never as a `dyn` object, so
its `async fn`s need no `async-trait`/boxing.

**8b. `ReleaseUpdate` accessors moved to a `UpdateConfig` supertrait.** The shared accessor methods
(`bin_name`, `current_version`, `target`, `release_tag`, `asset_identifier`, `auth_token`,
`bin_install_path`, `bin_path_in_archive`, the progress/transport/verify getters, `api_headers`, …)
now live on a new sealed supertrait `self_update::UpdateConfig`, and `ReleaseUpdate: UpdateConfig`
keeps only the three fetches plus `update`/`update_extended`. Calling an accessor on a
`Box<dyn ReleaseUpdate>` needs **no change** (trait-object method lookup sees the supertrait). The
only sites that need a fix are generic helpers bounded `R: ReleaseUpdate` that call an accessor
without naming the supertrait — add `use self_update::UpdateConfig;` (or bound
`R: ReleaseUpdate + UpdateConfig`, which is implied but makes the methods visible). Error:
`no method named `bin_name`/`current_version`/… found for type parameter`.

---

## Verify

Run the downstream crate's build/tests with its actual feature selection, e.g.:

```bash
cargo build
cargo test
```

A successful build with no `self_update`-related errors means the migration is complete.
