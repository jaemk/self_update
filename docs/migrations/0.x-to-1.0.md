# Migrating to `self_update` 1.0 (automation guide)

Scope: upgrading a downstream crate's usage of `self_update` from `0.x` to `1.0.0`.
Apply every rule below in order. Rules are either **mechanical** (deterministic
find/replace) or **manual** (needs local context - flag for review, do not blind-apply).
After applying, run the **Verify** step and resolve any remaining compiler errors using the
"Compiler-error -> fix" table.

---

## 0. Dependency

- File: `Cargo.toml`
- Mechanical: set the `self_update` dependency version requirement to `"1.0"` (or `"1"`).
- The crate is now edition 2024. The MSRV is raised to `1.88` (was `1.85`).

## 1. Builder setter renames (mechanical, method-call level)

Apply as literal substring replacements on Rust source (`.rs`) files. Each pattern includes
the leading `.` and trailing `(` so only method calls match (never field/type names). These
patterns match on method name only, not receiver type: if your code calls a same-named method
(e.g. `.timeout(`, `.header(`, `.show_progress(`) on a value that is **not** a `self_update`
builder or `Download`, scope the replacement to the `self_update` call sites named in each row's notes.

| # | find | replace | notes |
|---|------|---------|-------|
| 1.1 | `.with_url(` | `.api_base_url(` | github `Update`/`ReleaseList` builders. Takes the full API base URL, including any `/api/v3` path. A `.url(` on a github builder (an earlier 1.0 pre-release name) also becomes `.api_base_url(`. |
| 1.2 | `.with_host(` / `.instance_url(` / `.url(` | `.host(` | gitlab + gitea builders (the instance-base setter). gitea has no default host, so `.host(...)` is required: `build()` errors with `MissingField { field: "host" }` if missing (gitlab still defaults to `https://gitlab.com`). The `.instance_url(` and `.url(` forms cover earlier 1.0 pre-releases; scope `.url(` by backend (github -> `api_base_url`, gitlab/gitea -> `host`). |
| 1.3 | `.with_target(` | `.filter_target(` | the `ReleaseList` release-filter setter on all four backends. NOTE: the `Update` builder's own `.target(` (build-target triple, from the shared macro) is unchanged, do not rewrite it. Also covers a `.target(` on a `ReleaseList` builder from an earlier 1.0 pre-release. |
| 1.4 | `.access_key_id(` | `.access_key(` | s3 `Update` builder (`s3-auth`); the s3 `ReleaseList` builder already used `.access_key(` and is unchanged |
| 1.5 | `.set_headers(` | `.replace_headers(` | `Download`, replaces the whole `HeaderMap` (the additive header setter is now `request_header`, see 1.12) |
| 1.6 | `.show_progress(` | `.show_download_progress(` | `Download`, renamed to match the `Update` builder setter of the same name |
| 1.7 | `.target_version(` / `.target_version_tag(` | `.release_tag(` | `.target_version(` was the 0.x accessor call site; `.target_version_tag(` was the 0.x setter. Both collapse to `.release_tag(` (the `Update` builder setter and the `ReleaseUpdate` accessor now share the name; also see §3 for the accessor's return-type change). |
| 1.8 | `.identifier(` | `.asset_identifier(` | the asset-disambiguation setter on every `Update` builder and the `ReleaseUpdate` accessor; set and get share the name `asset_identifier`. |
| 1.9 | `.set_progress_callback(` | `.progress_callback(` | `Download` and every `Update` builder; the `Update` accessor is also `progress_callback` (set/get match). |
| 1.10 | `.set_progress_style(` | `.progress_style(` | `Download` and every `Update` builder. The argument shape also changed: it now takes one typed `ProgressStyle` instead of two strings, see §1c. |
| 1.11 | `.set_timeout(` | `.timeout(` | `Download`, matches the `Update`/`ReleaseList` builders' `.timeout(`. |
| 1.12 | `.set_header(` | `.request_header(` | `Download` additive single-header setter. Distinct from `replace_headers` (1.5), and named to match the `Update`/`ReleaseList` builders' `request_header`. Now takes `TryInto<HeaderName>`/`TryInto<HeaderValue>` and returns `&mut Self`, so string literals work, e.g. `.request_header("Accept", "application/octet-stream")`; an invalid header surfaces from `download_to` as `Error::InvalidHeader` (see 1a). |
| 1.13 | `.verifying_checksum(` / `.checksum(` | `.verify_checksum(` | every `Update` builder (`checksums` feature); matches the `verify_checksum()` accessor (set/get align). `.checksum(` covers an earlier 1.0 pre-release. |
| 1.14 | `.verify_keys(` | `.verifying_keys(` | every `Update` builder (`signatures` feature). The 0.x setter name `verifying_keys` is unchanged in 1.0; a 0.x user who only set keys needs no change. Rewrite any `.verify_keys(` call (whether a setter call from an earlier 1.0 pre-release or a READ-accessor call from 0.x or an earlier pre-release) to `verifying_keys`. Both setter and accessor are `verifying_keys` in 1.0. |

These renames are unambiguous: the old names no longer exist in 1.0, so any remaining
occurrence is a real call site to convert. The old names are not kept as `#[doc(alias)]`s (the
aliases were dropped, since they were never callable methods); use the canonical name.

## 1a. `request_header` now takes `TryInto` args (mechanical, non-breaking)

The `Update`/`ReleaseList` builder setter `.request_header(name, value)` now accepts anything
convertible to a header name/value, so a previously required `.parse().unwrap()` can be dropped:
`.request_header("X-Foo", "bar")` and
`.request_header(self_update::http::header::ACCEPT, "application/json")` both compile. An invalid
header name/value is no longer a panic at the call site - it is surfaced as an `Error::InvalidHeader`
from `build()` (see §2 for the structured `Error` variants). (Existing typed-argument call sites
still compile via the identity conversion.)

The standalone downloader's additive single-header setter is `Download::request_header` (renamed
from `set_header`, see row 1.12; it carries the `request_header` name to match the
`Update`/`ReleaseList` builders, not the bare `header` of an earlier 1.0 pre-release). It takes
`TryInto<HeaderName>`/`TryInto<HeaderValue>` and is infallible (returns `&mut Self`), so a string
value no longer needs `.parse()`: `.request_header("Accept", "application/octet-stream")` compiles
and chains. An invalid header is deferred and surfaced from `download_to` as
`Error::InvalidHeader`, matching the builders (an earlier 1.0 pre-release returned a `Result` from
the setter itself; drop any `?` on the call). Existing typed-argument call sites still compile via
the identity conversion. The old `set_header` name is not kept as a `#[doc(alias)]`; use
`request_header`.

## 1c. `progress_style` takes a typed `ProgressStyle` (mechanical, argument shape)

The `progress_style` setter on every `Update` builder and on `Download` takes a single
`self_update::ProgressStyle` value (under the `progress-bar` feature), not the two `impl Into<String>`
arguments (template, then chars) of an earlier 1.0 pre-release, so the two strings can't be
transposed. Build one with `ProgressStyle::new(template, chars)` (each `impl Into<String>`):
`.progress_style(self_update::ProgressStyle::new("[{bar:40}] {bytes}/{total_bytes}", "=>-"))`. The
type and setter are only present under `progress-bar` (see §5). Error:
`this method takes 1 argument but 2 arguments were supplied` at `progress_style`.

## 1b. String setters take `impl Into<String>` (mechanical, mostly non-breaking)

The string builder setters now take `impl Into<String>` instead of `&str`: `current_version`,
`release_tag`, `target`, `asset_identifier`, `bin_name`, `bin_path_in_archive`, `auth_token`,
and the backend setters `repo_owner` / `repo_name` / `api_base_url` / `host` / `filter_target` /
`bucket_name` / `asset_prefix` / `region`. String-literal call sites are unchanged. A site that
passed `&some_string` (a `&String`) no longer compiles against the `&str` parameter: drop the `&`
to move the `String` in, or pass `some_string.clone()` to keep your binding. Error:
`mismatched types: expected ... String, found &String` at the setter argument.

The utility constructors likewise take owned-or-borrowed arguments and dropped their lifetime
parameters:

- `Download::from_url(impl Into<String>)` (was `&str`).
- `Extract<'a>` -> `Extract`; `Extract::from_source(impl AsRef<Path>)` (was `&'a Path`), and
  `Extract::extract_into(impl AsRef<Path>)` / `Extract::extract_file(impl AsRef<Path>, ..)`
  (were `&Path`).
- `Move<'a>` -> `Move`; `Move::from_source(impl AsRef<Path>)`,
  `Move::replace_using_temp(impl AsRef<Path>)`, `Move::to_dest(impl AsRef<Path>)` (the last was
  `&Path`).
- `MoveAll<'a>` -> `MoveAll`; `MoveAll::from_temp(impl AsRef<Path>)`.

String-literal and `&Path` call sites are unchanged. The only edit is a named lifetime, e.g.
`let e: Extract<'_> = ...` -> `let e: Extract = ...`. Error:
`struct takes 0 lifetime arguments but 1 lifetime argument was supplied`.

## 2. Error enum (manual - semantic)

`self_update::errors::Error` is now `#[non_exhaustive]` and feature-specific variants were
collapsed. For every `match` (or `if let`) on an `Error` value:

- Replace arm patterns:
  - `Error::Reqwest(e)` -> `Error::Transport(e)`
  - `Error::Ureq(e)` -> `Error::Transport(e)`
  - `Error::StdTimeError(e)` -> `Error::S3Auth(e)`
  - `Error::TimeError(e)` -> `Error::S3Auth(e)`
  - `Error::Digest(e)` -> `Error::S3Auth(e)`
  - `Error::UrlParse(e)` -> `Error::S3Auth(e)`
- In `Error::Transport(e)` / `Error::S3Auth(e)`, `e` is now `Box<dyn std::error::Error + Send + Sync>`
  (was a concrete `reqwest::Error` / `ureq::Error` / etc.). If code called a concrete method on
  the inner error, downcast: `e.downcast_ref::<reqwest::Error>()`, or read it via
  `std::error::Error::source()` on the outer `Error`.
- Because `Error` is now `#[non_exhaustive]`, every external `match` on it MUST have a wildcard
  arm. If a `match` is now non-exhaustive, add `_ => { /* handle/ignore */ }`.
- Feature gating: `Error::S3Auth` only exists when the `s3-auth` feature is enabled (and
  `Error::Zip` / signature variants under their features); `Error::Transport` is always present. If
  the consumer does not enable `s3-auth`, omit the `S3Auth` arm: the required `_ =>` arm
  already covers it. Do NOT add an `Error::S3Auth` arm to a crate that doesn't enable `s3-auth`
  (it won't compile).

If two of the old arms collapse to the same new variant in one `match` (e.g. both `Reqwest`
and a later `Ureq` arm -> `Transport`), keep ONE `Error::Transport(e)` arm and delete the duplicate.

**`Error::Network(String)` is gone; non-2xx responses are now structured variants.** A request
that *completed* with a non-2xx status is no longer `Error::Network`. It is one of:
`Error::NotFound { url }` (404), `Error::Unauthorized { status, url }` (401/403), or
`Error::HttpStatus { status, url }` (any other non-2xx). A request that *could not complete*
(connection/TLS/timeout) is `Error::Transport(Box<dyn Error + Send + Sync>)`: the variant the
old `Error::Http` renamed to (the `From<reqwest::Error>` / `From<ureq::Error>` impls map here).
Both the `reqwest` and `ureq` clients now produce the same status variants (previously `ureq`
surfaced a non-2xx as `Error::Http`).

```rust
// before
match err {
    Error::Network(s) => log::error!("http status: {s}"),
    Error::Http(e)    => log::error!("transport: {e}"),
    _ => {}
}

// after (the struct variants are `#[non_exhaustive]`, so the patterns need `..`)
match err {
    Error::NotFound { url, .. }            => log::error!("not found: {url}"),
    Error::Unauthorized { status, url, .. } => log::error!("unauthorized {status}: {url}"),
    Error::HttpStatus { status, url, .. }   => log::error!("http status {status}: {url}"),
    Error::Transport(e)                     => log::error!("transport: {e}"),
    _ => {}
}
```

Code that only needs the status code can drop the per-variant arms and call
`err.http_status() -> Option<u16>` (404 for `NotFound`, the exact code for `Unauthorized` /
`HttpStatus`, `None` for any non-HTTP error).

**`Error::Zip`, `Error::Signature`, `Error::Json`, and `Error::SemVer` are now opaque.** Each wraps
`Box<dyn std::error::Error + Send + Sync>` instead of the concrete `zip::result::ZipError`
(`Error::Zip`) / `zipsign_api::ZipsignError` (`Error::Signature`, `signatures` feature) /
`serde_json::Error` (`Error::Json`) / `semver::Error` (`Error::SemVer`). The arm patterns still
match, but the bound `e` is now the box. Code that called a concrete method on the inner error must
inspect it via `Error::source()` on the outer `Error`, or downcast the box
(`e.downcast_ref::<zip::result::ZipError>()`).

**`Error::NonUTF8` is renamed to `Error::SignatureNonUTF8`** (`signatures` feature). Rename the arm
pattern.

**`Error::Update(String)` is gone, split into structured variants.** Replace arm patterns:

- a checksum digest mismatch -> `Error::ChecksumMismatch { expected, computed }` (both hex-encoded
  lowercase digests).
- a declined interactive confirmation prompt -> `Error::Aborted`.
- the post-update verify hook rejecting the binary -> `Error::VerificationRejected { reason: Option<String> }`
  (see §6a for the `verify_binary` callback that produces it).
- an internal invariant or task failure (e.g. a tokio blocking-task join) ->
  `Error::Internal { message: String, source: Option<Box<dyn std::error::Error + Send + Sync>> }`.
  `Internal` wraps the underlying error as `source` when it has one, surfaced via `Error::source()`.

Any `match` that branched on `Error::Update` must switch to the matching variant above.

**`Error::Config(String)` is gone, split into structured variants.** Replace arm patterns:

- a missing required field -> `Error::MissingField { field: &'static str }` (e.g. `"repo_owner"`,
  `"bin_name"`, `"region"`, `"host"` on gitea).
- an invalid request/download header -> `Error::InvalidHeader { source }`.
- an invalid auth token -> `Error::InvalidAuthToken { source }`.
- a root-certificate or client-build failure -> `Error::InvalidCertificate { source }` (from
  `build()` / `download_to`).
- a bad progress-bar template -> `Error::InvalidProgressStyle { source }` (`progress-bar` feature).
- the s3-auth SigV4 host-extraction failure -> `Error::S3Auth`.

No residual `Error::Config` remains; the variant is removed entirely. `InvalidHeader` /
`InvalidAuthToken` / `InvalidCertificate` / `InvalidProgressStyle` carry
`source: Box<dyn std::error::Error + Send + Sync>` and chain through `Error::source()`.

**`Error::Release(String)` is gone, split into structured variants.** Replace arm patterns:

- no matching release/asset -> `Error::NoReleaseFound { target: Option<String> }` (the requested
  target triple when the lookup was asset-scoped, else `None`).
- a payload missing a field -> `Error::MissingAssetField { field: &'static str }`.
- an unparseable response -> `Error::InvalidResponse { source }` (chains through `Error::source()`).
  A malformed (non-array) release-listing body maps to `InvalidResponse`, not `NoReleaseFound`.

**New in 1.0: `Error::CompressionNotEnabled`.** Returned when a gzip asset is detected but the
`compression-tar-gz` feature is off (0.x installed the still-compressed bytes as the binary). The
feature is in the 1.0 default set (see §5.2), so only `default-features = false` builds can hit it.

The new struct variants are each `#[non_exhaustive]`, as are `Error::Unauthorized` and
`Error::HttpStatus`: destructure them with a trailing `..`, e.g.
`Error::Unauthorized { status, .. }`. `Error::Io` still wraps a concrete `std::io::Error` (its
`kind()` is reachable). Errors that wrap an underlying cause (`InvalidResponse`, `InvalidHeader`,
`InvalidAuthToken`, `InvalidCertificate`, `InvalidProgressStyle`, `Internal` with a source) now
return it from `std::error::Error::source()`; code that walked `source()` and previously got
`None` for these will now see the wrapped cause.

**Display strings changed for two variants.** `Error::ArchiveNotEnabled` now renders with the
`"ArchiveNotEnabledError: ..."` prefix and `Error::SignatureNonUTF8` with `"SignatureError: ..."`,
matching the `<Name>Error:` prefix of every other variant. Display strings are human-facing and may
change between releases: match on variants or use `err.http_status()` / `err.url()` for
programmatic decisions, do not parse the Display output.

**Also `#[non_exhaustive]` in 1.0:** `VersionStatus` (renamed from `Status`, see §2a),
`ArchiveKind`, and `Compression` are now also `#[non_exhaustive]`. An exhaustive `match` on any of
them, especially `VersionStatus` (returned by every `update()` call), must gain a `_ =>` arm. The
compiler will flag any non-exhaustive pattern. The `Download`, `Extract`, `Move`, and `MoveAll`
utility structs are also `#[non_exhaustive]` (their fields are already private, so this only blocks
a future struct-literal construction), as are each backend's concrete `Update` struct (returned by `build()`), each backend's
`AsyncUpdate` wrapper (e.g. `github::AsyncUpdate`, returned by `build_async()`), and
`custom::AsyncUpdate` (returned by `custom::build_async()`).

## 2a. Status enum renames + predicate renames (mechanical)

The two update-result enums are renamed:
- `Status` (the lightweight result of `update()`, carrying a version string) is now
  `VersionStatus`.
- `UpdateStatus` (the extended result of `update_extended()`, carrying a `Release`) is now
  `ReleaseStatus`.
- The method `UpdateStatus::into_status(current_version)` is now
  `ReleaseStatus::into_version_status(current_version)`.

Both are re-exported at the crate root under the new names (`self_update::VersionStatus`,
`self_update::ReleaseStatus`). The variants (`UpToDate` / `Updated`) and the accessor methods
(`version`, `updated_release`, `into_updated_release`) are unchanged. Rename the types at every
mention: `self_update::Status` -> `self_update::VersionStatus`, `self_update::UpdateStatus` ->
`self_update::ReleaseStatus`, `.into_status(` -> `.into_version_status(`.

The predicates were also renamed to a matching pair: `.uptodate(` -> `.is_up_to_date(` and
`.updated(` -> `.is_updated(`, on both enums. The old names no longer exist, so any remaining call
is a real site to convert.

## 3. `ReleaseUpdate` accessor return types (manual - type)

The `ReleaseUpdate` trait accessors now return borrows:
- `current_version`, `target`, `bin_name`, `bin_path_in_archive`, `progress_template`,
  `progress_chars` return `&str` (were `String`).
- `release_tag` (renamed from `target_version`/`target_version_tag`, see §1.7),
  `asset_identifier` (renamed from `identifier`, see §1.8), `auth_token`
  return `Option<&str>` (were `Option<String>`).
- `bin_install_path` returns `&std::path::Path` (was an owned `PathBuf`). Only relevant if you
  named the return type; append `.to_path_buf()` where you need ownership.
- `verifying_keys` (`signatures` feature) returns `&[VerifyingKey]` (was `Vec<VerifyingKey>` under the old `verify_keys` accessor name). The accessor was renamed from `verify_keys` to `verifying_keys`; both setter and accessor now share the name `verifying_keys` (see §1.14).

Fix sites where ownership is required: append `.to_string()` / `.map(str::to_owned)`.
- `let v: String = updater.current_version();` -> `let v: String = updater.current_version().to_string();`
- `let id: Option<String> = updater.identifier();` -> `... = updater.asset_identifier().map(str::to_owned);`
Sites that only read (`==`, `format!`, passing `&str`) need no change.

## 3a. Release-fetch methods return `Releases`; `build()` returns the concrete `Update` (manual - type)

The release-listing surface now returns a new `Releases` type instead of a bare
`Release` / `Vec<Release>`, the filtered fetch is renamed, and the availability check changed
shape:

- `get_latest_release()` returns `Result<Releases>` (was `Result<Release>`).
- The filtered list fetch is renamed `get_latest_releases` -> `get_newer_releases`. It returns
  `Result<Releases>` holding only releases strictly newer than the current version, and no longer
  takes a `current_version` argument (it reads the configured current version). Was
  `get_latest_releases(&self, current_version: &str) -> Result<Vec<Release>>`.
- `get_release_version(&self, ver: &str) -> Result<Release>` is unchanged (specific-tag lookup).
- `is_update_available()` stays on the updater but now returns `Result<Option<Release>>` (the
  newest strictly-newer release, `None` when up to date; was `Result<bool>` and fetched the list
  a second time). Error: `mismatched types: expected bool, found Option<Release>`.
- Under the `async` feature, the built updater adds `get_latest_release_async()` and
  `get_newer_releases_async()` returning `Result<Releases>`, and `get_release_version_async()`
  returning `Result<Release>`. `is_update_available_async()` exists with the same new shape as
  the sync verb: `Result<Option<Release>>`.

`build()` returns the concrete backend `Update` struct (was `Box<dyn ReleaseUpdate>`). The update
verbs (`update`, `update_extended`, `get_latest_release`, `get_newer_releases`,
`get_release_version`, `is_update_available`) are inherent methods, so `.build()?.update()?`
needs no `use ... ReleaseUpdate;` import. `Update` is `Send`. A binding typed
`Box<dyn ReleaseUpdate>` drops the annotation or names the concrete type
(`backends::<name>::Update`). Error: `mismatched types: expected Box<dyn ReleaseUpdate>` at
`build()`.

`Releases` (re-exported at the crate root, `self_update::Releases`) carries the fetched releases
plus the current version and exposes `all() -> &[Release]`, `latest() -> Option<&Release>` (newest
first), `into_vec() -> Vec<Release>`, and `is_update_available() -> Result<bool>`. It is distinct
from the per-backend `ReleaseList` builder; do not conflate the two.

The per-backend `ReleaseList::fetch` (github/gitlab/gitea/s3) also returns `Result<Releases>` (was
`Result<Vec<Release>>`), so the standalone listing surface matches the updater's, and gains an
async sibling `fetch_async` under the `async` feature. Recover the old `Vec<Release>` with
`.into_vec()`: `ReleaseList::configure()./*..*/.build()?.fetch()?.into_vec()`. A bare listing
carries no current version, so `Releases::current_version()` is `None` for a `fetch()` result.
`Releases::from_releases(..)` and `Releases::from_listing(..)` are constructors for building a
`Releases` in downstream tests (the latter is the bare-listing state). Error:
`mismatched types: expected `Vec<Release>`, found `Releases`` at the `fetch()` call.

Rewrite call sites:
- A pre-check is now one fetch: `updater.is_update_available()?.is_some()` (sync) or
  `updater.get_latest_release_async().await?.is_update_available()?` (async). Any earlier
  `updater.is_update_available()` call site that expected a `bool` must be rewritten.
- `let r: Release = updater.get_latest_release()?;` -> `... = updater.get_latest_release()?;` now
  yields a `Releases`; read the newest via `.latest()` or take the list with `.into_vec()`.
- `updater.get_latest_releases("1.2.3")?` -> `updater.get_newer_releases()?` (rename and drop the
  argument), then call `.all()` / `.into_vec()` for the `Vec<Release>` you previously got. NOTE:
  this rename applies to the updater only; the custom-source method is separately renamed
  `get_latest_releases` -> `get_releases` (see §9b).

## 3b. `Release` / `ReleaseAsset` fields are private; use getters (mechanical)

`Release` and `ReleaseAsset` fields are no longer public. Replace field access with the getter of
the same name (each returns a borrow):

| # | old field access | new getter | type |
|---|------------------|------------|------|
| 3b.1 | `release.name` | `release.name()` | `&str` |
| 3b.2 | `release.version` | `release.version()` | `&str` |
| 3b.3 | `release.date` | `release.date()` | `&str` |
| 3b.4 | `release.body` | `release.body()` | `Option<&str>` (was `Option<String>`) |
| 3b.5 | `release.assets` | `release.assets()` | `&[ReleaseAsset]` |
| 3b.6 | `asset.name` | `asset.name()` | `&str` |
| 3b.7 | `asset.download_url` | `asset.download_url()` | `&str` |

Append `.to_string()` / `.to_owned()` where you need ownership; `.body()` returns `Option<&str>`,
so use `.map(str::to_owned)` for an owned `Option<String>`. Construction is unchanged:
`Release::builder()` / `ReleaseAsset::new(..)` build these `#[non_exhaustive]` types, and there was
no public struct-literal path. Error: `field `name` of struct `Release` is private`.

## 4. `ReleaseUpdate` is sealed (manual - rare)

`impl self_update::update::ReleaseUpdate for MyType { .. }` no longer compiles (the trait is
sealed). There is no mechanical fix: rework the custom backend onto the public `ReleaseSource` (or
`AsyncReleaseSource`) trait and drive it through `backends::custom` (see §9). Flag for human
review. Note that `build()` on the built-in backends now returns the concrete `Update`, not
`Box<dyn ReleaseUpdate>` (see §3a); the trait remains callable and usable as a generic bound.

## 5. Feature flags (manual - `Cargo.toml`)

**5.1 Two features were renamed (mechanical).** Update any `features = [...]` list and any
`self_update?/<feature>` reference:

| old feature | new feature | notes |
|-------------|-------------|-------|
| `default-tls` | `native-tls` | the native-TLS backend (forwards `reqwest?/native-tls` / `ureq?/native-tls`). |
| `compression-flate2` | `compression-tar-gz` | gzip-in-tar support (`archive-tar` + `flate2` + `either`). |

Error: `the package ... does not contain this feature: default-tls` / `compression-flate2`.

**5.2 The default feature set changed (manual).** It went from `["reqwest", "default-tls"]` to
`["reqwest", "rustls", "progress-bar", "github", "archive-tar", "compression-tar-gz"]`:

- **Default TLS is now `rustls`, not native-TLS.** A default build no longer links the system
  OpenSSL/native-TLS stack. If you require native-TLS, set `default-features = false` and enable
  `native-tls` (plus a client and any backends you use). No code change; the TLS choice is
  build-time only.
- **Backends are now per-feature.** Only `github` is in the default set. The `gitlab`, `gitea`,
  and `s3` backends are opt-in: enable the matching feature to compile
  `self_update::backends::<gitlab|gitea|s3>`. If your code uses one, add its feature, e.g.
  `features = ["gitlab"]`. Error: `could not find `gitlab` in `backends``.
- **`s3-auth` now implies `s3`.** Enabling `s3-auth` automatically enables `s3` (it no longer
  needs both). The `s3` feature also gates the `quick-xml` dependency. The old no-op `s3` alias
  feature from the 0.x line is gone; the `s3` feature now genuinely gates the backend.
- **tar/gzip archive support is on by default.** `archive-tar` and `compression-tar-gz` are in
  the default set, so the common `.tar.gz` release asset works without the explicit
  `archive-tar compression-flate2` features 0.x required. Zip stays opt-in (`archive-zip` +
  `compression-zip-deflate`/`compression-zip-bzip2`). On a `default-features = false` build
  without `compression-tar-gz`, a gzip asset now fails with `Error::CompressionNotEnabled` (see
  §2) instead of installing the still-compressed bytes.

```toml
# A consumer that used the gitlab backend on a default 0.x install:
# before: gitlab compiled unconditionally
self_update = "0.39"
# after: gitlab is opt-in
self_update = { version = "1", features = ["gitlab"] }
```

**5.3 The terminal progress bar is behind a `progress-bar` feature (manual).** The default-on
`progress-bar` feature gates the `indicatif` dependency. The byte-level `progress_callback` and the
`show_download_progress` toggle stay always available (when `progress-bar` is off, the bar is a
no-op). The `progress_style` setter and the `ProgressStyle` type (see §1c) are only present under
`progress-bar`. A default build is unaffected; a `default-features = false` build that wants the bar
should add `progress-bar`, or omit it to drop `indicatif`.

**5.4 HTTP clients and TLS backends now coexist (manual).** Enabling both `reqwest` and `ureq`, or
both `native-tls` and `rustls`, is no longer a `compile_error!`:

- `reqwest` + `ureq` together both compile; the sync API prefers `reqwest` when both are on.
- `native-tls` + `rustls` together both compile; the per-call builders prefer `rustls`.

`cargo build --all-features` builds. The only surviving guards are: at least one of `reqwest` /
`ureq` must be enabled (`no HTTP client selected`), and `async` requires `reqwest` (which `async`
already pulls in). No action is required, but you may simplify a manifest that set
`default-features = false` solely to dodge a now-removed conflict. If a `default-features = false`
manifest selected `ureq` or `rustls`, it still needs a client enabled, but no longer needs to drop
the other client/TLS backend to avoid a collision.

## 5a. s3 endpoint/region validation (manual - runtime)

The s3 `Update` and `ReleaseList` `build()` now validate the endpoint/region pairing. The `S3`,
`S3DualStack`, and `DigitalOceanSpaces` endpoints require a `region`, so a missing region is now
an `Error::MissingField { field: "region" }` from `build()` instead of surfacing later from the
first request. `GCS` and `Generic` are unaffected. No code change is needed if you already set
`.region(..)` for those endpoints; otherwise add it, or handle the earlier `Error::MissingField`
from `build()`. A named
constructor `backends::s3::AccessKey::new(id, secret)` is also available alongside the existing
tuple `From` impls.

## 6. Removed / privatised items (manual - rare)

- **s3 `auth_token` is removed:** the s3 `Update` and `ReleaseList` builders no longer have an
  `.auth_token(..)` setter. The s3 backend authenticates via `access_key` (AWS SigV4), never a
  bearer token, so the setter never had any effect there. Delete any leftover `.auth_token(..)` on
  an s3 builder, and if private-bucket auth was intended, use `.access_key((id, secret))` (the
  `s3-auth` feature). This applies only to the s3 backend; `auth_token` is a real setter on
  github/gitlab/gitea. Error: `no method named `auth_token` ... s3 ... builder`.
- **Dropped re-exports:** `self_update::self_replace` and `self_update::TempDir` are no
  longer re-exported. If you referenced `self_update::self_replace::...` or `self_update::TempDir`,
  add the `tempfile` and/or `self-replace` crates as your own dependencies and import them
  directly (`tempfile::TempDir`, `self_replace::self_replace`). The `http`, `reqwest`/`ureq`, and
  `zipsign_api` re-exports are **unchanged**.
- **s3 `AccessKey` is now public and `#[non_exhaustive]`:** the credential type is
  re-exported as `self_update::backends::s3::AccessKey` (under `s3-auth`). Build one through
  `AccessKey::new(id, secret)` or its `From<(&str,&str)>` / `From<(String,String)>` impls (i.e.
  pass an `(id, secret)` tuple to `.access_key(...)`), not a struct literal: `#[non_exhaustive]`
  leaves room for an additive field (e.g. an STS session token) later. No change needed unless you
  named the type.
- `self_update::GetArchiveReaderResult` is no longer public. It had no consumer use; delete
  any reference.
- `std::error::Error::description` is no longer implemented on `Error` (it was deprecated);
  use `Display` / `source()`.
- `self_update::should_update(current, latest)` is removed (deprecated since 0.4.2). Replace
  with `self_update::version::bump_is_greater(current, latest)` (same "is latest newer"
  semantics), or `version::bump_is_compatible(current, latest)` for a compatibility check.
- **Behavior change (Linux, `native-tls`):** the crate no longer sets `SSL_CERT_FILE` /
  `SSL_CERT_DIR` for you before each request (it used `std::env::set_var`, unsound in a
  multi-threaded process and hardcoded to Debian/Ubuntu paths). No code change is needed, but if
  you relied on that auto-setting in a minimal environment (some containers, `musl` static builds)
  and now hit a certificate-verification error, export those variables yourself, or build with the
  `rustls` feature (the default). See the crate "Troubleshooting" docs.

## 6a. Post-update verify hook renamed, now returns `Result<()>` (manual)

The builder setter `verify_with` is renamed to `verify_binary`, and the callback's return type
changed from `bool` to `Result<()>`:

- old: `.verify_with(|new_exe: &Path| -> bool { ... })` (`true` accepts, `false` rejects)
- new: `.verify_binary(|new_exe: &Path| -> self_update::Result<()> { ... })` (`Ok(())` accepts,
  `Err(..)` rejects)

A rejection now produces `Error::VerificationRejected` (see §2). Return `Ok(())` where the closure
returned `true`, and `Err(...)` where it returned `false`:

```rust
.verify_binary(|new_exe| run_version_check(new_exe))   // already returns Result<()>
// or, mapping a bool:
.verify_binary(|new_exe| if ok(new_exe) { Ok(()) } else {
    Err(self_update::Error::verification_rejected("self-check failed"))
})
```

`Error::verification_rejected(reason)` builds the rejection directly and is surfaced as-is; any
other error returned from the hook is wrapped in a `VerificationRejected` whose `reason` is that
error's message.

Errors: `no method named `verify_with``; `expected ... bool, found ... Result` at the verify closure.

## 6b. s3 `EndPoint` -> `Endpoint`, `Generic` is now a tuple variant (manual)

The s3 endpoint enum was renamed `EndPoint` -> `Endpoint`, and its `Generic` variant changed from a
struct variant `{ end_point: String }` to a tuple variant `(String)`. The
`ReleaseListBuilder`/`UpdateBuilder` setter `end_point(..)` is renamed `endpoint(..)`:

```rust
// before
use self_update::backends::s3::EndPoint;
builder.end_point(EndPoint::Generic { end_point: "https://s3.example.com/bucket/".into() });
// after
use self_update::backends::s3::Endpoint;
builder.endpoint(Endpoint::Generic("https://s3.example.com/bucket/".into()));
```

The `From<&str>` / `From<String>` impls (`"...".into()` -> `Endpoint::Generic`) are unchanged, so a
call that passed a string into `.endpoint(..)` only needs the setter rename. `Endpoint` is still
`#[non_exhaustive]`. Errors: `cannot find type `EndPoint``; `no method named `end_point``;
`variant `Generic` ... does not have a field named `end_point``.

## 6c. `Download` per-client setters removed (manual)

`Download::reqwest_client`, `Download::reqwest_async_client`, and `Download::ureq_agent` are no
longer public. To run a download through a custom client, configure the client on the `Update`
builder (`reqwest_client` / `ureq_agent` / `reqwest_async_client`, or the new
`http_client(Arc<dyn HttpClient>)`, see §9) and let the update flow forward it to the download.
These per-client setters remain on the `Update`/`ReleaseList` builders. Error:
`no method named `reqwest_client` ... Download`.

## 7. Optional cleanups (non-breaking, safe to skip)

- The example header type `reqwest::header::ACCEPT` can become `self_update::http::header::ACCEPT`
  (the `http` crate is re-exported as `self_update::http`; no separate `http` dep needed).
- `[u8; zipsign_api::PUBLIC_KEY_LENGTH]` can be written `self_update::VerifyingKey` (signatures
  feature); the `zipsign_api` crate is re-exported as `self_update::zipsign_api`.
- `self_update::ReleaseUpdate` is now re-exported at the crate root (was
  `self_update::update::ReleaseUpdate`).
- `ReleaseStatus` (renamed from `UpdateStatus`, see §2a) gained
  `updated_release() -> Option<&Release>` and `into_updated_release() -> Option<Release>` to read
  the installed release without a `match` (which `#[non_exhaustive]` would force a wildcard arm
  onto).
- Every backend `Update`/`ReleaseList` builder gained `.unattended()`, which sets `no_confirm(true)`
  and `show_output(false)` in one call for daemon/CI use. Without it the default
  (`no_confirm == false`) blocks on stdin for the interactive confirmation prompt.
- `ArchiveKind` now implements `Display` (rendering `tar.gz` / `zip` / `tar` / `gz` / `plain`);
  the `Error::NoSignatures` message uses it in place of the old `Debug` form.
- `ReleaseAsset`'s field order is now `name, download_url` (matching
  `ReleaseAsset::new(name, download_url)`). Positional / `Debug` ordering shifted, but the fields are
  now private getters (see §3b), so read them via `.name()` / `.download_url()`.
- **Inject any HTTP client.** The `Update`/`ReleaseList` builders have
  `http_client(Arc<dyn HttpClient>)` (and `http_client_async(Arc<dyn AsyncHttpClient>)` under
  `async`): pass any object-safe client (a test double, a wrapper, your application's client), not
  just reqwest/ureq. Implement `self_update::http_client::HttpClient` (and `HttpResponse`) for your
  own transport; the public `http_client` module exposes these traits. The existing
  `reqwest_client` / `ureq_agent` / `reqwest_async_client` setters are now thin wrappers over it.
  A custom sync `HttpResponse` implements only `headers()` and `body()` (the crate parses JSON/XML
  from the body reader itself).
- **Custom root CAs.** `self_update::Certificate` (`from_pem` / `from_der`) plus
  `add_root_certificate` on the `Update`/`ReleaseList` builders and on `Download` trust a
  private/internal CA without injecting a whole pre-built client. A malformed certificate is
  `Error::InvalidCertificate` from `build()` / `download_to`.
- **Auth-host allow list.** `allow_auth_host(host)` on the builders authorizes an asset CDN/mirror
  host to receive the auth token; `dangerously_allow_non_https_auth_forwarding()` lifts the https
  requirement for a host-matched request (see §8).
- `AsyncReleaseUpdate` is re-exported at the crate root under `async`: a public sealed trait
  mirroring `ReleaseUpdate` for the async verbs, usable as a generic bound.
- `self_update::version::cmp_versions(a, b) -> Result<Ordering>`: a total-order semver comparison.
- `ReleaseStatus::version() -> Option<&str>` mirrors `VersionStatus::version` (`None` on the
  up-to-date arm).
- `Releases::from_releases(..)` and `Releases::from_listing(..)` build a `Releases` value in your
  own tests.

## 8. Behavior changes (no API change)

These change runtime behavior, not signatures. No code edit is required, but review if you depended
on the old behavior:

- **Auth token is host-scoped.** An `auth_token` is now applied to *both* the release-listing and
  the binary-download requests (previously download auth was github-specific), but only to
  requests whose host matches the backend's configured API host (or an `allow_auth_host` entry),
  over https. A server-supplied asset `download_url` or pagination `Link` pointing at a different
  host does not receive the token. Use `allow_auth_host(host)` to authorize an asset CDN/mirror
  host, and `dangerously_allow_non_https_auth_forwarding()` to allow the token over http to a
  host-matched request. The scheme is per backend: `Authorization: token <token>` for
  github/gitea, `Authorization: Bearer <token>` for gitlab. A user-set `Authorization` via
  `request_header` overrides the backend default on both paths.
- **git listing filters per-item.** During `update()`, releases on each page that are not strictly
  newer than the configured current version are filtered out per-item; pagination continues
  through all pages regardless. `ReleaseList::fetch` walks every page unfiltered.
- **s3 continuation pagination.** The s3 listing now follows `NextContinuationToken`, so buckets
  with more than one page of objects list fully (previously capped at 100 keys). New `max_keys`
  (a `u16`, clamped `1..=1000`, default 1000) and `signature_ttl` (`s3-auth`, default 300s)
  setters replace the hardcoded cap and presigned-URL TTL.
- **Configurable retry backoff.** A new `retry_backoff(base, max)` setter on the
  `Update`/`ReleaseList` builders configures the exponential backoff (default 100ms base, ~3.2s
  cap). The `retries` budget now also covers the binary download's request-establishment phase
  (before any bytes stream); a mid-stream failure is still not retried.

## 9. Custom-backend async + accessor split (manual - rare)

Three changes to the custom-backend / `ReleaseUpdate` surface (the async source under the `async`
feature):

**9a. `ReleaseSource` is sync-only.** The trait is three sync fetches plus `Send + Sync`, with no
`Self: Clone` bound and no defaulted `*_async` helpers. If you implemented or called any
`get_*_async` default on `ReleaseSource`:

- For a **natively-async** source, implement the
  [`AsyncReleaseSource`](https://docs.rs/self_update/latest/self_update/trait.AsyncReleaseSource.html)
  trait instead: the same three fetches with clean names (no `_async` suffix:
  `get_latest_release`, `get_releases`, `get_release_version`), and drive the update with
  `self_update::backends::custom::AsyncUpdate::configure().source(..).build_async()?.update_async().await?`
  (mirrors a built-in backend's `build_async()`). The trait requires its returned futures to be
  `Send` (the methods return `impl Future<Output = ...> + Send`), enforced at the impl site rather
  than failing later at the spawn site. A plain `async fn` body satisfies this as long as it holds
  no non-`Send` value across an `.await`; if it does, the impl no longer compiles.
- To reuse an existing **`Clone` sync `ReleaseSource`** from the async API, wrap it:
  `AsyncUpdate::configure().source(self_update::backends::custom::Blocking::new(my_sync_source))`.
  `Blocking` runs the sync fetches on `tokio::task::spawn_blocking`. Its inner field is private;
  construct it with `Blocking::new(source)` and read the wrapped source back via `into_inner()` /
  `as_inner()`.

`AsyncReleaseSource` is consumed through generics (`AsyncUpdate<S>`), never as a `dyn` object, so
its futures need no `async-trait`/boxing.

**9b. `get_latest_releases` is renamed `get_releases` and drops its `current_version` argument.**
On both `ReleaseSource` and `AsyncReleaseSource`:

```rust
// before
fn get_latest_releases(&self, current_version: &str) -> self_update::Result<Vec<Release>> { .. }
// after
fn get_releases(&self) -> self_update::Result<Vec<Release>> { .. }
```

Rename the method in your `impl` and drop the parameter (the argument was unused on this path; the
current version is carried by the updater, not the source), and update any direct call. Errors:
`not all trait items implemented, missing: `get_releases`` /
`method `get_latest_releases` is not a member of trait `ReleaseSource``.

NOTE: this is a different method from the updater/`ReleaseUpdate` filtered fetch, which was renamed
`get_newer_releases` (§3a). The source method returns the unfiltered candidate list (newest-first);
the updater method returns the strictly-newer subset.

A source builds the `#[non_exhaustive]` release-flow error variants via the public constructors
`Error::no_release_found()` / `Error::no_release_found_for_target(target)`,
`Error::missing_asset_field(field)` (takes `impl Into<String>`),
`Error::invalid_response(source)`, `Error::http_status_error(status, url)`, and
`Error::checksum_mismatch(expected, computed)` (struct-literal construction fails downstream).

**9c. `ReleaseUpdate` accessors moved to a `UpdateConfig` supertrait.** The shared accessor methods
(`bin_name`, `current_version`, `target`, `release_tag`, `asset_identifier`, `auth_token`,
`bin_install_path`, `bin_path_in_archive`, the progress/transport/verify getters, `api_headers`,
...) now live on a new sealed supertrait `self_update::UpdateConfig`, and `ReleaseUpdate: UpdateConfig`
keeps only the fetches plus `update`/`update_extended`. The update verbs are inherent on the
concrete `Update` that `build()` returns (§3a), so plain update chains need no import. Reading an
accessor on a concrete `Update` value needs `use self_update::UpdateConfig;` in scope. A generic
helper bounded `R: ReleaseUpdate`, or a call through a `dyn ReleaseUpdate`, needs no import: the
`R: ReleaseUpdate` bound (and the trait object's own trait) already brings the supertrait accessors
into scope. Error: `no method named `bin_name`/`current_version`/... found` for the concrete
`Update` value when the trait is not in scope.
