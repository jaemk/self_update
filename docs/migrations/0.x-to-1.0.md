# Migrating to `self_update` 1.0 (automation guide)

Scope: upgrading a downstream crate's usage of `self_update` from `0.x` to `1.0.0`.
Apply every rule below in order. Rules are either **mechanical** (deterministic
find/replace) or **manual** (needs local context — flag for review, do not blind-apply).
After applying, run the **Verify** step and resolve any remaining compiler errors using the
"Compiler-error → fix" table.

---

## 0. Dependency

- File: `Cargo.toml`
- Mechanical: set the `self_update` dependency version requirement to `"1.0"` (or `"1"`).

## 1. Builder setter renames (mechanical, method-call level)

Apply as literal substring replacements on Rust source (`.rs`) files. Each pattern includes
the leading `.` and trailing `(` so only method calls match (never field/type names).

| # | find | replace | notes |
|---|------|---------|-------|
| 1.1 | `.with_url(` | `.url(` | github `Update`/`ReleaseList` builders |
| 1.2 | `.with_host(` | `.url(` | gitlab + gitea builders (the custom-endpoint setter). gitea has **no default host**, so `.url(...)` is required — `build()` now errors with `` `url` required `` if it is missing (gitlab still defaults to `https://gitlab.com`). |
| 1.3 | `.with_target(` | `.target(` | `ReleaseList` builders, all backends |
| 1.4 | `.access_key_id(` | `.access_key(` | s3 `Update` builder (`s3-auth`); the s3 `ReleaseList` builder already used `.access_key(` and is unchanged |
| 1.5 | `.set_headers(` | `.replace_headers(` | `Download` — renamed to signal it replaces the whole `HeaderMap` (the additive `set_header` is unchanged) |
| 1.6 | `.show_progress(` | `.show_download_progress(` | `Download` — renamed to match the `Update` builder setter of the same name |
| 1.7 | `.target_version(` | `.target_version_tag(` | `ReleaseUpdate` accessor — renamed to match the `target_version_tag` builder setter (also see §3 for its return-type change). Does not match the already-correct `.target_version_tag(` setter calls. |

These renames are unambiguous: the old names no longer exist in 1.0, so any remaining
occurrence is a real call site to convert.

## 2. Error enum (manual — semantic)

`self_update::errors::Error` is now `#[non_exhaustive]` and feature-specific variants were
collapsed. For every `match` (or `if let`) on an `Error` value:

- Replace arm patterns:
  - `Error::Reqwest(e)` → `Error::Http(e)`
  - `Error::Ureq(e)` → `Error::Http(e)`
  - `Error::StdTimeError(e)` → `Error::S3Auth(e)`
  - `Error::TimeError(e)` → `Error::S3Auth(e)`
  - `Error::Digest(e)` → `Error::S3Auth(e)`
  - `Error::UrlParse(e)` → `Error::S3Auth(e)`
- In `Error::Http(e)` / `Error::S3Auth(e)`, `e` is now `Box<dyn std::error::Error + Send + Sync>`
  (was a concrete `reqwest::Error` / `ureq::Error` / etc.). If code called a concrete method on
  the inner error, downcast: `e.downcast_ref::<reqwest::Error>()`, or read it via
  `std::error::Error::source()` on the outer `Error`.
- Because `Error` is now `#[non_exhaustive]`, every external `match` on it MUST have a wildcard
  arm. If a `match` is now non-exhaustive, add `_ => { /* handle/ignore */ }`.
- Feature gating: `Error::S3Auth` only exists when the `s3-auth` feature is enabled (and
  `Error::Zip` / signature variants under their features); `Error::Http` is always present. If
  the consumer does not enable `s3-auth`, omit the `S3Auth` arm — the required `_ =>` arm
  already covers it. Do NOT add an `Error::S3Auth` arm to a crate that doesn't enable `s3-auth`
  (it won't compile).

If two of the old arms collapse to the same new variant in one `match` (e.g. both `Reqwest`
and a later `Ureq` arm → `Http`), keep ONE `Error::Http(e)` arm and delete the duplicate.

## 3. `ReleaseUpdate` accessor return types (manual — type)

The `ReleaseUpdate` trait accessors now return borrows:
- `current_version`, `target`, `bin_name`, `bin_path_in_archive`, `progress_template`,
  `progress_chars` return `&str` (were `String`).
- `target_version_tag` (renamed from `target_version` — see §1.7), `identifier`, `auth_token`
  return `Option<&str>` (were `Option<String>`).

Fix sites where ownership is required: append `.to_string()` / `.map(str::to_owned)`.
- `let v: String = updater.current_version();` → `let v: String = updater.current_version().to_string();`
- `let id: Option<String> = updater.identifier();` → `... = updater.identifier().map(str::to_owned);`
Sites that only read (`==`, `format!`, passing `&str`) need no change.

## 4. `ReleaseUpdate` is sealed (manual — rare)

`impl self_update::update::ReleaseUpdate for MyType { .. }` no longer compiles (the trait is
sealed). There is no mechanical fix: a custom backend must be reworked to use a built-in
backend, or the upgrade blocked pending an upstream feature request. Flag for human review.

## 5. Feature flags (manual — `Cargo.toml`)

Invalid HTTP-client / TLS feature combinations are now a hard `compile_error!`:
- `reqwest` + `ureq` together → invalid. Pick one. For `ureq`, set
  `default-features = false` and enable `ureq` (+ one TLS feature).
- *neither* `reqwest` *nor* `ureq` → invalid (`no HTTP client selected`). This fires when you
  set `default-features = false` (e.g. to switch to `ureq` or `rustls`) but forget to enable a
  client; enable exactly one of `reqwest` or `ureq`.
- `default-tls` + `rustls` together → invalid. For `rustls`, set `default-features = false`.

If the manifest enables `self_update` with `features = ["ureq"]` or `["rustls"]` WITHOUT
`default-features = false`, add `default-features = false` and ensure exactly one client
(`reqwest` xor `ureq`) and one TLS backend (`default-tls` xor `rustls`).

## 6. Removed / privatised items (manual — rare)

- `self_update::GetArchiveReaderResult` is no longer public. It had no consumer use; delete
  any reference.
- `std::error::Error::description` is no longer implemented on `Error` (it was deprecated);
  use `Display` / `source()`.
- `self_update::should_update(current, latest)` is removed (deprecated since 0.4.2). Replace
  with `self_update::version::bump_is_greater(current, latest)` (same "is latest newer"
  semantics), or `version::bump_is_compatible(current, latest)` for a compatibility check.
- **Behavior change (Linux, `default-tls`):** the crate no longer sets `SSL_CERT_FILE` /
  `SSL_CERT_DIR` for you before each request (it used `std::env::set_var`, unsound in a
  multi-threaded process and hardcoded to Debian/Ubuntu paths). No code change is needed, but if
  you relied on that auto-setting in a minimal environment (some containers, `musl` static builds)
  and now hit a certificate-verification error, export those variables yourself, or build with the
  `rustls` feature. See the crate "Troubleshooting" docs.

## 7. Optional cleanups (non-breaking, safe to skip)

- The example header type `reqwest::header::ACCEPT` can become `self_update::http::header::ACCEPT`
  (the `http` crate is re-exported as `self_update::http`; no separate `http` dep needed).
- `[u8; zipsign_api::PUBLIC_KEY_LENGTH]` can be written `self_update::VerifyingKey` (signatures
  feature); the `zipsign_api` crate is re-exported as `self_update::zipsign_api`.
- `self_update::ReleaseUpdate` is now re-exported at the crate root (was
  `self_update::update::ReleaseUpdate`).
- s3 `.auth_token(` is now `#[deprecated]` (a compiler **warning**, not an error) and a no-op:
  the s3 backend authenticates via `access_key` (AWS SigV4), never a bearer token, so the call
  never did anything. Delete the `.auth_token(..)` call on s3 `Update` builders; if private-bucket
  auth was intended, replace it with `.access_key((id, secret))` (the `s3-auth` feature). This
  applies only to the **s3** backend — `auth_token` remains functional on github/gitlab/gitea.

---

## Verify

Run the downstream crate's build/tests with its actual feature selection, e.g.:

```bash
cargo build
cargo test
```

A successful build with no `self_update`-related errors means the migration is complete.

## Compiler-error → fix

| compiler error (substring) | rule |
|---|---|
| `no method named `with_url`` / `with_host` / `with_target` | §1.1–1.3 |
| `no method named `access_key_id`` | §1.4 |
| `no variant named `Reqwest`/`Ureq`/`StdTimeError`/`TimeError`/`Digest`/`UrlParse`` | §2 |
| `non-exhaustive patterns` on `Error` | §2 (add `_ =>` arm) |
| `mismatched types: expected `String`, found `&str`` near an accessor | §3 |
| `the trait `ReleaseUpdate` cannot be implemented` / sealed-trait error | §4 |
| `features ... are mutually exclusive` (compile_error) | §5 |
| `no HTTP client selected` (compile_error) | §5 |
| `use of deprecated ... `auth_token`` (warning, s3 builder) | §7 |
| `cannot find type `GetArchiveReaderResult`` | §6 |
| `cannot find function `should_update`` | §6 |
