# Migrating to `self_update` 1.0

`self_update` 1.0 is the first stable release. It makes a focused set of breaking changes to
clean up the public API before committing to long-term `1.x` stability. This was the only
window in which they could land. Most upgrades are small and mechanical; the sections below
walk through each change, why it was made, and exactly what to do.

> Looking for a terse, copy-paste / automation-friendly checklist? See
> [`0.x-to-1.0.md`](0.x-to-1.0.md).

---

## 1. Builder method names are unified

We removed the `with_` prefix and made the four backends speak the same vocabulary, so you no
longer have to remember which backend calls a thing what.

| Before | After | Where |
|--------|-------|-------|
| `.with_url("https://...")` | `.api_base_url("https://...")` | github (the full API base URL, including any `/api/v3` path) |
| `.with_host("https://...")` | `.host("https://...")` | gitlab, gitea (the instance base URL) |
| `.with_target("x86_64-...")` | `.filter_target("x86_64-...")` | `ReleaseList` on all backends (the release filter) |
| `.access_key_id((id, secret))` | `.access_key((id, secret))` | s3 (`s3-auth`) |
| `.target_version_tag("v1.2.3")` | `.release_tag("v1.2.3")` | every `Update` builder |
| `.identifier("musl")` | `.asset_identifier("musl")` | every `Update` builder |
| `.verifying_checksum(c)` | `.verify_checksum(c)` | every `Update` builder (`checksums`) |
| `.set_progress_callback(cb)` | `.progress_callback(cb)` | `Download` + every `Update` builder |
| `.set_progress_style(t, c)` | `.progress_style(ProgressStyle::new(t, c))` | `Download` + every `Update` builder (now a typed value, see below) |
| `.set_timeout(d)` | `.timeout(d)` | `Download` |
| `.set_header(n, v)` | `.request_header(n, v)` | `Download` (single header; `replace_headers` still replaces the whole map) |
| `.show_progress(b)` | `.show_download_progress(b)` | `Download` |
| `.set_headers(map)` | `.replace_headers(map)` | `Download` |
| `.verify_with(\|p\| -> bool)` | `.verify_binary(\|p\| -> Result<()>)` | every `Update` builder (see section 7) |

The signature-key setter `verifying_keys(..)` (`signatures` feature) keeps its 0.x name; its
read accessor is now also `verifying_keys()` (renamed from `verify_keys()`, see section 3).

**Why:** the names were tidied so that setters and accessors match (e.g. the `release_tag` /
`asset_identifier` setter now reads back under the same name), so the standalone `Download`
utility's setters line up with the `Update` builder's (`timeout`, `progress_callback`,
`progress_style`, `show_download_progress`), and so each backend's endpoint setter says what it
takes: github's is `api_base_url(...)` (the full API base URL, including any `/api/v3` path) and
gitlab/gitea's is `host(...)` (the instance base URL), with the `ReleaseList` release filter named
`filter_target` (distinct from the build-target `target` on the `Update` builder). These were pure
papercuts that a 1.0 is the right time to fix. The old names are not kept as `#[doc(alias)]`s (the
aliases were never callable methods, only a rustdoc-search footgun); use the canonical name.

While here, `request_header` now accepts plain strings as well as typed header values:
`.request_header("X-Foo", "bar")` works, and an invalid header is reported as an
`Error::InvalidHeader` from `build()` instead of panicking at the call site. The standalone
downloader's single-header setter is now `Download::request_header` (matching the builders' verb)
and also takes string literals. It is infallible (returns `&mut Self`); an invalid header is
deferred and surfaced from `download_to` as `Error::InvalidHeader`, matching the builders:
`download.request_header("Accept", "application/octet-stream")`.

`progress_style` now takes one typed `ProgressStyle` value instead of two strings (template, then
chars), so they can't be transposed:
`.progress_style(self_update::ProgressStyle::new("[{bar:40}] {bytes}/{total_bytes}", "=>-"))`. The
`ProgressStyle` type and the setter live under the `progress-bar` feature (see section 5).

The string builder setters now take `impl Into<String>` instead of `&str`: `current_version`,
`release_tag`, `target`, `asset_identifier`, `bin_name`, `bin_path_in_archive`, `auth_token`, and
the backend setters `repo_owner` / `repo_name` / `api_base_url` / `host` / `filter_target` /
`bucket_name` / `asset_prefix` / `region`. String literals keep compiling. The only thing that changes is a site
that passed `&some_string` (a `&String`): drop the `&` to move the `String` in, or pass
`some_string.clone()` to keep your binding.

**What to do:** find-and-replace the calls. The old names are gone, so the compiler points you
at every site (`no method named with_url ...`).

```rust
// before
let releases = self_update::backends::gitlab::ReleaseList::configure()
    .with_host("https://gitlab.example.com")
    .repo_owner("me").repo_name("app")
    .with_target("x86_64-unknown-linux-gnu")
    .build()?;

// after
let releases = self_update::backends::gitlab::ReleaseList::configure()
    .host("https://gitlab.example.com")
    .repo_owner("me").repo_name("app")
    .filter_target("x86_64-unknown-linux-gnu")
    .build()?;
```

## 2. The `Error` enum changed shape

`self_update::errors::Error` is now `#[non_exhaustive]`, the feature-specific variants were
replaced by two opaque ones, and the HTTP error model was restructured:

- `Error::Reqwest(..)` and `Error::Ureq(..)` -> **`Error::Transport(Box<dyn Error + Send + Sync>)`**
  (a request that could not complete: connection, TLS, timeout). This is the variant the old
  `Error::Http` was renamed to.
- `Error::StdTimeError`, `Error::TimeError`, `Error::Digest`, `Error::UrlParse` (all the
  `s3-auth` signing errors) -> **`Error::S3Auth(Box<dyn Error + Send + Sync>)`**
- A *completed* non-2xx response is no longer `Error::Network(String)`. It is one of three
  structured variants: **`Error::NotFound { url }`** (404),
  **`Error::Unauthorized { status, url }`** (401/403), and
  **`Error::HttpStatus { status, url }`** (any other non-2xx). Both the `reqwest` and `ureq`
  clients now produce the same variants (previously `ureq` reported a non-2xx as `Error::Http`),
  so a consumer can tell release-not-found from auth failure from other statuses regardless of
  client. `Error::http_status() -> Option<u16>` returns the code when you only need that.

**Why:** previously an exhaustive `match` on `Error` only compiled for one HTTP-client feature
set: code written for the default `reqwest` build broke if a user switched to `ureq`, and vice
versa, because the variants were `#[cfg]`-gated. Collapsing them into opaque variants makes your
error handling independent of which client/features are enabled. Marking the enum
`#[non_exhaustive]` means we can add error categories in future `1.x` releases without it being
a breaking change.

**What to do:**

```rust
// before
match err {
    Error::Reqwest(e) => log::error!("http: {e}"),
    Error::Network(s) => log::error!("status: {s}"),
    Error::Release(s) => log::error!("release: {s}"),
    // ...one arm per variant...
}

// after (the struct variants are `#[non_exhaustive]`, so the patterns need `..`)
match err {
    Error::Transport(e)                     => log::error!("transport: {e}"),
    Error::NotFound { url, .. }             => log::error!("not found: {url}"),
    Error::Unauthorized { status, url, .. } => log::error!("unauthorized {status}: {url}"),
    Error::HttpStatus { status, url, .. }   => log::error!("status {status}: {url}"),
    Error::NoReleaseFound { target, .. }    => log::error!("no release: {target:?}"),
    Error::MissingAssetField { field, .. }  => log::error!("missing field: {field}"),
    Error::InvalidResponse { source, .. }   => log::error!("bad response: {source}"),
    _ => log::error!("other: {err}"), // required: the enum is now non_exhaustive
}
```

If you only need the status code, skip the per-variant arms and call `err.http_status()`, which
is `Some(404)` for `NotFound`, the exact code for `Unauthorized` / `HttpStatus`, and `None` for
any non-HTTP error.

Four more variants became opaque the same way: `Error::Zip`, `Error::Signature` (the latter under
the `signatures` feature), `Error::Json`, and `Error::SemVer` now wrap `Box<dyn Error + Send + Sync>`
instead of the concrete `zip::result::ZipError` / `zipsign_api::ZipsignError` / `serde_json::Error` /
`semver::Error`. The arm patterns still match, but you can no longer call an inner-error method on the
bound value directly; inspect it via `source()` or downcast (same as below). And the signatures-only
`Error::NonUTF8` variant is renamed to `Error::SignatureNonUTF8` (rename the arm).

The three stringly-typed catch-all variants `Error::Config(String)`, `Error::Release(String)`, and
`Error::Update(String)` are each split into structured variants you can branch on directly:

| Old variant | New variants |
|-------------|--------------|
| `Error::Config(String)` | `Error::MissingField { field }`, `Error::InvalidHeader { source }`, `Error::InvalidAuthToken { source }`, `Error::InvalidCertificate { source }`, `Error::InvalidProgressStyle { source }` (`progress-bar`) |
| `Error::Release(String)` | `Error::NoReleaseFound { target }`, `Error::MissingAssetField { field }`, `Error::InvalidResponse { source }` |
| `Error::Update(String)` | `Error::ChecksumMismatch { expected, computed }`, `Error::Aborted`, `Error::VerificationRejected { reason }`, `Error::Internal { message, source }` |

A few details:

- `MissingField.field` and `MissingAssetField.field` are `&'static str` (e.g. `"repo_owner"`,
  `"region"`, `"bin_name"`).
- `NoReleaseFound.target` is `Option<String>` (the requested target triple, or `None`).
- `ChecksumMismatch` carries the hex-encoded lowercase `expected` / `computed` digests, `Aborted` is
  a declined interactive confirmation, and `VerificationRejected` is the post-update verify hook
  rejecting the binary (see section 7). `Internal` is an internal invariant or task failure (e.g. a
  tokio blocking-task join).
- `InvalidHeader`, `InvalidAuthToken`, `InvalidResponse`, `InvalidCertificate`,
  `InvalidProgressStyle`, and `Internal` (when it wraps one) carry a boxed `source` that now chains
  through `std::error::Error::source()`. If you walked `source()` before and got `None` for one of
  these, you will now see the cause.
- Nothing of `Error::Config` remains. A root-certificate or client-build failure is
  `Error::InvalidCertificate { source }` (from `build()` / `download_to`), a bad progress-bar
  template is `Error::InvalidProgressStyle { source }` (`progress-bar` feature), and the s3-auth
  SigV4 host-extraction failure maps to `Error::S3Auth`.
- A malformed (non-array) release-listing body maps to `Error::InvalidResponse`, not
  `NoReleaseFound`.
- New `Error::CompressionNotEnabled`: a gzip asset was selected but `compression-tar-gz` is off.
  0.x installed the still-compressed bytes as the binary; 1.0 errors instead. The feature is in
  the 1.0 default set (see section 5), so this only fires on `default-features = false` builds.
- The new struct variants are each `#[non_exhaustive]` (as are `Unauthorized` / `HttpStatus`), so
  destructure them with a trailing `..`: `Error::Unauthorized { status, .. }`.
- `Error::Io` still wraps a concrete `std::io::Error`, so its `kind()` is reachable.

Two Display strings also changed: `Error::ArchiveNotEnabled` now renders with the
`"ArchiveNotEnabledError: ..."` prefix and `Error::SignatureNonUTF8` with `"SignatureError: ..."`,
matching the `<Name>Error:` prefix of every other variant. Display strings are human-facing and may
change between releases, so match on variants or use `err.http_status()` / `err.url()` for
programmatic decisions rather than parsing the Display output.

If you need the concrete underlying error, downcast the box or use the standard `source()`:

```rust
if let Error::Transport(e) = &err {
    if let Some(re) = e.downcast_ref::<reqwest::Error>() { /* ... */ }
}
```

> **Note:** `Error::S3Auth` only exists when you enable the `s3-auth` feature; `Error::Transport`
> is always present. If you don't use `s3-auth`, just leave out the `S3Auth` arm: the
> wildcard `_ =>` arm covers it. (Adding an `Error::S3Auth` arm without the feature won't
> compile.)

> **Note:** In 1.0, `VersionStatus` (renamed from `Status`, see below), `ArchiveKind`, and
> `Compression` also became `#[non_exhaustive]`. If you have an exhaustive `match` on any of them
> (especially `VersionStatus`, returned by every `update()` call), add a `_ =>` arm: the compiler
> will flag any exhaustive match that no longer covers all variants. The `Download`, `Extract`,
> `Move`, and `MoveAll` utility structs are also `#[non_exhaustive]` now (their fields were already
> private, so this only blocks a future struct-literal construction), as are each backend's
> concrete `Update` struct (returned by `build()`), each backend's `AsyncUpdate` wrapper
> (e.g. `github::AsyncUpdate`, returned by `build_async()`), and `custom::AsyncUpdate`
> (returned by `custom::build_async()`).

> **Note:** the two update-result enums were renamed. `Status` (the result of `update()`, carrying
> a version string) is now `VersionStatus`, and `UpdateStatus` (the result of `update_extended()`,
> carrying a `Release`) is now `ReleaseStatus`; the method `UpdateStatus::into_status(..)` is now
> `ReleaseStatus::into_version_status(..)`. Both are still re-exported at the crate root. Their
> predicates were also renamed for a consistent pair: `uptodate()` to `is_up_to_date()` and
> `updated()` to `is_updated()`. Rename the types, the method, and the predicate calls; the old
> names no longer exist, so the compiler points at each one.

## 3. `ReleaseUpdate` accessors return borrows

The accessor methods on the `ReleaseUpdate` trait now return `&str` / `Option<&str>` instead of
owned `String` / `Option<String>` (`current_version`, `target`, `bin_name`,
`bin_path_in_archive`, `progress_template`, `progress_chars`, `release_tag`, `asset_identifier`,
`auth_token`). `bin_install_path` returns `&Path` instead of an owned `PathBuf`. The `verifying_keys`
accessor (`signatures` feature, renamed from `verify_keys`) similarly returns `&[VerifyingKey]`
instead of `Vec<VerifyingKey>`; setter and accessor now share the name `verifying_keys`.

**Why:** these are simple field reads; returning owned `String`s forced an allocation on every
call for no reason. This is the kind of signature that can't be fixed after 1.0, so it had to
happen now.

The `target_version` accessor was also **renamed to `release_tag`** and `identifier` to
`asset_identifier`, each matching its builder setter exactly (so set and get share a name). Same
window-of-opportunity reasoning: a name can't change after 1.0. The standalone `Download`
utility's setters were likewise lined up with the `Update` builder: `set_headers` ->
`replace_headers` (it replaces the whole header map), `show_progress` -> `show_download_progress`,
`set_timeout` -> `timeout`, `set_header` -> `request_header`, `set_progress_callback` ->
`progress_callback`, and `set_progress_style` -> `progress_style`.

**What to do:** most call sites (`==`, `format!`, passing a `&str`) need no change. Where you
specifically need ownership, add `.to_string()`:

```rust
let v: String = updater.current_version().to_string(); // was: updater.current_version()
```

`Release` and `ReleaseAsset` also stopped exposing public fields; read them through getters of the
same name (each returns a borrow):

| Before | After | Type |
|--------|-------|------|
| `release.name` | `release.name()` | `&str` |
| `release.version` | `release.version()` | `&str` |
| `release.date` | `release.date()` | `&str` |
| `release.body` | `release.body()` | `Option<&str>` |
| `release.assets` | `release.assets()` | `&[ReleaseAsset]` |
| `asset.name` | `asset.name()` | `&str` |
| `asset.download_url` | `asset.download_url()` | `&str` |

```rust
// before
let url = &release.assets[0].download_url;
// after
let url = release.assets()[0].download_url();
```

Where you need an owned value, add `.to_string()` (or `.map(str::to_owned)` for `.body()`'s
`Option<&str>`). The fields are `Arc<str>` internally now, so the getters are cheap. Construction is
unchanged: `Release::builder()` and `ReleaseAsset::new(..)` already build these `#[non_exhaustive]`
types, and there was no public struct-literal path.

## 4. `build()` returns a concrete `Update`; `ReleaseUpdate` is a sealed trait

Every backend's `build()` now returns the concrete backend `Update` struct instead of a
`Box<dyn ReleaseUpdate>`. The update verbs (`update`, `update_extended`, `get_latest_release`,
`get_newer_releases`, `get_release_version`, `is_update_available`) are inherent methods on it, so
`.build()?.update()?` needs no trait import. `Update` is `Send`, so it can move to a worker
thread. A binding typed `Box<dyn ReleaseUpdate>` can drop the annotation (or name the concrete
type, e.g. `backends::github::Update`).

`ReleaseUpdate` still exists as a sealed trait, usable as a generic bound: you can call it, but
you can no longer `impl ReleaseUpdate for YourOwnType`.

**Why:** the trait was always an internal plumbing abstraction, never meant as an extension
point. Sealing it lets us evolve it in future `1.x` releases without breaking anyone. (It's also
why the `&str` change in §3 was safe to make.) Returning the concrete type drops a needless
allocation and makes the returned updater `Send`.

**What to do:** almost nobody implements this trait. If you did (a custom backend), implement
the public `ReleaseSource` (or `AsyncReleaseSource`) trait instead and drive it through
`backends::custom` (section 10).

## 5. Features and the build baseline

The crate is now edition 2024 and the MSRV is raised to `1.88` (from `1.85`). Several
feature-level changes affect your `Cargo.toml`.

**Two features were renamed:**

| Before | After |
|--------|-------|
| `default-tls` | `native-tls` |
| `compression-flate2` | `compression-tar-gz` |

```toml
# before
self_update = { version = "0.39", default-features = false, features = ["ureq", "default-tls"] }
# after
self_update = { version = "1", default-features = false, features = ["ureq", "native-tls"] }
```

**The default TLS backend is now `rustls`, not native-TLS.** The default feature set is
`["reqwest", "rustls", "progress-bar", "github", "archive-tar", "compression-tar-gz"]`. A default
build no longer links the system OpenSSL stack. If you need native-TLS, set
`default-features = false` and enable `native-tls` explicitly (plus a client and the backends you
use).

**tar/gzip archive support is on by default.** `archive-tar` and `compression-tar-gz` are in the
default set, so the common `.tar.gz` release asset works out of the box (in 0.x you had to enable
`archive-tar compression-flate2` yourself). Zip stays opt-in (`archive-zip` +
`compression-zip-deflate`/`compression-zip-bzip2`). On a `default-features = false` build without
`compression-tar-gz`, a gzip asset now fails with `Error::CompressionNotEnabled` instead of
installing the still-compressed bytes (see section 2).

**Backends are now opt-in.** Only `github` is on by default. The `gitlab`, `gitea`, and `s3`
backends each sit behind a feature of the same name. If your code uses
`self_update::backends::gitlab` (or gitea/s3), enable that feature:

```toml
self_update = { version = "1", features = ["gitlab"] }
```

`s3-auth` now implies `s3` (you no longer enable both), and the `s3` feature also gates the
`quick-xml` dependency. The old no-op `s3` alias feature from the 0.x line is gone; `s3` now
genuinely gates the backend.

**The terminal progress bar is behind a `progress-bar` feature** (default-on), which gates the
`indicatif` dependency. The byte-level `progress_callback` and the `show_download_progress` toggle
stay always available; when `progress-bar` is off, the bar is a no-op. The `progress_style` setter
and the `ProgressStyle` type are only present under `progress-bar`. A default build is unaffected; a
`default-features = false` build that wants the bar should add `progress-bar`, or omit it to drop
`indicatif`.

**HTTP clients and TLS backends now coexist.** Earlier work made enabling both `reqwest` and `ureq`,
or both `native-tls` and `rustls`, a hard compile error. Those guards are gone:

- `reqwest` + `ureq` can both be compiled; the sync API prefers `reqwest` when both are on.
- `native-tls` + `rustls` can both be compiled; the per-call builders prefer `rustls`.

`cargo build --all-features` now builds. The only rules left are that at least one client must be
enabled (`no HTTP client selected`), and `async` requires `reqwest` (which it already pulls in). The
transport is now a runtime trait object (`HttpClient`) rather than a compile-time-selected function,
so the two clients no longer collide at link time. You may simplify a manifest that set
`default-features = false` only to dodge a now-removed conflict; a `default-features = false`
manifest selecting `ureq` or `rustls` still needs a client enabled, but no longer needs to drop the
other client/TLS backend.

```toml
# ureq + native-tls, no progress bar
self_update = { version = "1", default-features = false, features = ["ureq", "native-tls", "github"] }
```

Alongside this, the `Update`/`ReleaseList` builders gained an `http_client(Arc<dyn HttpClient>)`
setter (and `http_client_async` under `async`) so you can inject any object-safe client, not just
reqwest or ureq. The existing `reqwest_client` / `ureq_agent` / `reqwest_async_client` setters still
work; they are now thin wrappers over it. See the optional section at the end for implementing your
own transport.

## 6. Small removals

- **Two convenience re-exports were dropped:** `self_update::self_replace` and
  `self_update::TempDir`. They pinned downstream code to the crate's exact `tempfile` /
  `self-replace` versions for no real benefit. If you used them, add `tempfile` and/or
  `self-replace` to your own `Cargo.toml` and import them directly (`tempfile::TempDir`,
  `self_replace::self_replace`). The genuinely useful re-exports (`http`, the selected client
  (`reqwest` / `ureq`), and `zipsign_api`) are unchanged.
- `self_update::GetArchiveReaderResult` is no longer public: it was the return type of a
  private function and had no usable purpose. Delete any reference (there almost certainly
  isn't one).
- The deprecated `Error::description()` method is gone (it had been deprecated in std since
  Rust 1.27). Use `Display` (`{err}`) or `err.source()`.
- `should_update(current, latest)` is gone (deprecated since 0.4.2). It was a thin
  version comparison; use `self_update::version::bump_is_greater(current, latest)` for the
  same "is `latest` newer than `current`" check, or `version::bump_is_compatible(..)` if you
  want a semver-compatibility check.
- **s3 `.auth_token(..)` is removed.** The s3 `Update` and `ReleaseList` builders no longer have an
  `.auth_token(..)` setter. The s3 backend signs requests with an `access_key` (AWS SigV4); it never
  used a bearer token, so `auth_token` was always a no-op there. Delete any leftover `.auth_token(..)`
  on an s3 builder, and if you meant to authenticate to a private bucket, use `.access_key((id, secret))`
  (the `s3-auth` feature) instead. This only affects the s3 backend; `auth_token` is a real setter on
  github/gitlab/gitea.
- **s3 `build()` validates the endpoint/region pairing.** The `S3`, `S3DualStack`, and
  `DigitalOceanSpaces` endpoints embed the region in their hostname, so a missing `region` is now an
  `Error::MissingField` from `build()` rather than a failure on the first request. `GCS` and
  `Generic` are unaffected (they carry no region in the URL). Set `.region(..)` for those three
  endpoints if you weren't already.

## 7. Checking for an update returns a `Releases` value

The release-listing methods on the updater now hand back a `Releases` value instead of a bare
`Release` / `Vec<Release>`, the filtered list fetch is renamed, and the availability check
changed shape:

- `get_latest_release()` returns `Releases` (was `Release`).
- `get_latest_releases(current_version)` is now `get_newer_releases()`: no argument (it reads the
  version you configured), returns a `Releases` holding only releases strictly newer than the
  current version.
- `get_release_version(tag)` (specific-tag lookup) is unchanged.
- `is_update_available()` now returns `Result<Option<Release>>`: the newest strictly-newer
  release, or `None` when up to date (was a `bool` that fetched the release list a second time).
  `is_update_available_async()` has the same new shape on the async updaters.

`Releases` (re-exported as `self_update::Releases`) holds the fetched releases plus your current
version. It has `all() -> &[Release]`, `latest() -> Option<&Release>` (newest first),
`into_vec() -> Vec<Release>`, and `is_update_available() -> Result<bool>`. It is a different type
from the per-backend `ReleaseList` builder.

The per-backend `ReleaseList::fetch` (github/gitlab/gitea/s3) also returns `Releases` now instead of
`Vec<Release>`, so the standalone listing surface matches the updater's, and gains an async sibling
`fetch_async` under the `async` feature. Recover the old vector with `.into_vec()`:

```rust
// before
let releases: Vec<Release> = ReleaseList::configure()./*..*/.build()?.fetch()?;
// after
let releases: Vec<Release> = ReleaseList::configure()./*..*/.build()?.fetch()?.into_vec();
```

A bare listing has no current version, so `Releases::current_version()` is `None` for a `fetch()`
result. `Releases::from_releases(..)` and `Releases::from_listing(..)` are constructors for
building a `Releases` in your own tests (the latter is the bare-listing state).

**Why:** the old `is_update_available()` fetched the release list, then `update()` fetched it again.
The new shape is a single fetch, and the `Option<Release>` tells you what you would update to.

**What to do:** a light "is there a newer release?" pre-check is one fetch and one call:

```rust
let updater = self_update::backends::github::Update::configure()
    .repo_owner("me").repo_name("app").bin_name("app")
    .current_version(cargo_crate_version!())
    .build()?;

if updater.is_update_available()?.is_some() {
    updater.update()?;
}
```

Under the `async` feature the updater has `get_latest_release_async()` /
`get_newer_releases_async()` (returning `Releases`) and `get_release_version_async()` (returning
`Release`); the async pre-check is
`updater.get_latest_release_async().await?.is_update_available()?`. Where you previously read the
newest release from `get_latest_release()`, call `.latest()` on the returned `Releases`; where you
took the whole list from `get_latest_releases(..)`, call `.all()` or `.into_vec()` on
`get_newer_releases()`'s result.

## 8. The post-update verify hook

The `verify_with` builder setter is renamed to `verify_binary`, and its callback now returns
`Result<()>` instead of `bool`:

```rust
// before
.verify_with(|new_exe: &Path| -> bool { run_check(new_exe).is_ok() })
// after
.verify_binary(|new_exe: &Path| -> self_update::Result<()> { run_check(new_exe) })
```

Return `Ok(())` to accept the freshly-extracted binary and `Err(..)` to reject it. A rejection now
produces `Error::VerificationRejected` (see section 2) rather than a generic `Error::Update`, so the
failure reason can be carried. A `bool` could only say "rejected", not why. Build the rejection with
`Error::verification_rejected("reason")` (surfaced as-is); any other error returned from the hook is
wrapped with its message as the reason.

## 9. The s3 endpoint type

`EndPoint` is renamed to `Endpoint`, its `Generic` variant is now a tuple, and the setter
`end_point` is now `endpoint`:

```rust
// before
use self_update::backends::s3::EndPoint;
builder.end_point(EndPoint::Generic { end_point: "https://s3.example.com/bucket/".into() });
// after
use self_update::backends::s3::Endpoint;
builder.endpoint(Endpoint::Generic("https://s3.example.com/bucket/".into()));
```

The `"...".into()` -> `Endpoint::Generic` conversions still work, so a call that passed a string only
needs the `end_point` -> `endpoint` rename.

## 10. Custom backends: a clean sync/async split

If you write a **custom backend** (you `impl ReleaseSource`), a few things changed. Most people who
only use the built-in github/gitlab/gitea/s3 backends can skip this section.

**`ReleaseSource` is sync.** It is three plain sync fetch methods (and no `Clone` requirement), with
no defaulted `*_async` helpers. The sync and async worlds are mirror images of each other, just like
the built-in backends' `build()` vs `build_async()`: `build()` returns the concrete blocking
`Update`; `build_async()` returns a per-backend `AsyncUpdate` wrapper (e.g. `github::AsyncUpdate`)
that exposes only the `*_async` verbs as inherent methods (no `use self_update::AsyncReleaseUpdate`
import needed), so calling a blocking `.update()` on it is a compile error.

- **Native async source:** implement the `AsyncReleaseSource` trait, the same three fetches, with
  clean names (`get_latest_release`, `get_releases`, `get_release_version`, **no** `_async`
  suffix). Drive it with the async custom updater:

  ```rust,ignore
  use self_update::backends::custom::AsyncUpdate;
  let status = AsyncUpdate::configure()
      .source(MyAsyncHost)
      .bin_name("app")
      .current_version("1.0.0")
      .build_async()?
      .update_async().await?;
  ```

  There's no `async-trait` involved: the updater is generic over your source, so the futures cost
  nothing extra. The trait requires its returned futures to be `Send` (the methods return
  `impl Future<Output = ...> + Send`), so a non-`Send` impl fails to compile at the impl site instead
  of later at the spawn site. A plain `async fn` body satisfies this unless it holds a non-`Send`
  value across an `.await`.

- **Reuse a sync source from async code:** if you already have a `Clone` sync `ReleaseSource` and
  just want to call it from a tokio app, wrap it in `Blocking`:

  ```rust,ignore
  use self_update::backends::custom::{AsyncUpdate, Blocking};
  AsyncUpdate::configure().source(Blocking::new(my_sync_source)) /* ... */;
  ```

  `Blocking` runs your sync fetches on `tokio::task::spawn_blocking`. Its inner field is private:
  construct it with `Blocking::new(source)` and read the wrapped source back with `into_inner()` /
  `as_inner()`.

**`get_latest_releases` is renamed `get_releases` and drops its `current_version` argument.** On
both `ReleaseSource` and `AsyncReleaseSource`:

```rust
// before
fn get_latest_releases(&self, current_version: &str) -> Result<Vec<Release>> { .. }
// after
fn get_releases(&self) -> Result<Vec<Release>> { .. }
```

The argument was unused on this path; the current version is carried by the updater, not the
source. The new name says what the method returns: the source's full unfiltered candidate list
(newest-first), not just the "latest" ones. It is the *updater's* filtered fetch that was renamed
to `get_newer_releases` (section 7) — the two are different methods with different semantics.
There is also a new public `AsyncReleaseUpdate` trait (re-exported at the crate root under `async`),
a sealed mirror of `ReleaseUpdate` for the async verbs, usable as a bound.

**Error constructors for source impls.** The release-flow `Error` variants are
`#[non_exhaustive]`, so a downstream source cannot build them as struct literals. Use the public
constructors instead: `Error::no_release_found()` (or `Error::no_release_found_for_target(target)`
when an asset-scoped lookup failed), `Error::missing_asset_field(field)` (any `Into<String>`,
so dynamic paths like `format!("assets[{i}].url")` work), `Error::invalid_response(source)`,
`Error::http_status_error(status, url)`, and `Error::checksum_mismatch(expected, computed)`.

**Accessors moved to a `UpdateConfig` supertrait.** The many getter methods on `ReleaseUpdate`
(`bin_name`, `current_version`, `target`, ...) now live on a new supertrait
`self_update::UpdateConfig` (and `ReleaseUpdate: UpdateConfig`). The update verbs are inherent on
the concrete `Update` (section 4), so a plain `.build()?.update()?` chain needs no import; calling
an accessor on a concrete `Update` value needs `use self_update::UpdateConfig;` in scope. A generic
helper written `fn f<R: ReleaseUpdate>(u: &R)` that calls one needs no import: the `R: ReleaseUpdate`
bound already brings the supertrait accessors into scope. Likewise a call through a
`dyn ReleaseUpdate` needs no import.

## 11. Behavior changes (no code edit needed)

These change runtime behavior rather than any signature. Review them if you depended on the old
behavior:

- **Auth is host-scoped.** An `auth_token` is now sent on *both* the listing and the
  binary-download requests (not just listing), but only to requests whose host matches the
  backend's configured API host, over https. A server-supplied asset `download_url` or pagination
  `Link` pointing at a different host does not receive the token. If your assets are served from
  a CDN or mirror that legitimately needs it, authorize that host with `allow_auth_host(host)`;
  `dangerously_allow_non_https_auth_forwarding()` lifts the https requirement for a host-matched
  request (cleartext credential, internal networks only). The scheme follows the backend:
  `Authorization: token <token>` for github/gitea, `Bearer <token>` for gitlab. A user-set
  `Authorization` via `request_header` overrides the default on both paths.
- **git listing filters per-item.** During `update()`, releases on each page that are not
  strictly newer than your current version are filtered out per-item; pagination continues through
  all pages regardless. `ReleaseList::fetch` walks every page unfiltered.
- **s3 lists fully.** The s3 listing now follows `NextContinuationToken`, so a bucket with more than
  one page of objects is fully listed (previously capped at 100 keys). New `max_keys` (a `u16`,
  clamped `1..=1000`) and `signature_ttl` (`s3-auth`) setters replace the hardcoded cap and
  presigned-URL TTL.
- **Configurable retry backoff.** A new `retry_backoff(base, max)` setter tunes the exponential
  backoff (default 100ms base, ~3.2s cap). The `retries` budget now also covers the download's
  request-establishment phase (before any bytes stream); a mid-stream failure is still not retried.

## Niceties you can now adopt (optional, non-breaking)

These aren't required, but 1.0 makes a few things pleasanter:

- **`http` is re-exported.** Header constants for `Download::request_header` are reachable as
  `self_update::http::header::ACCEPT`, no need to add the `http` crate yourself, and it works the
  same under `reqwest` or `ureq`.
- **Inject any HTTP client.** Implement `self_update::http_client::HttpClient` (and `HttpResponse`)
  for your own transport and pass it with `http_client(Arc<dyn HttpClient>)` on the
  `Update`/`ReleaseList` builders (and `http_client_async` under `async`), including a test double
  that needs no network. The public `http_client` module exposes these traits. A custom
  `HttpResponse` implements only `headers()` and `body()`; the crate parses JSON/XML from the body
  reader itself.
- **Custom root CAs without a custom client.** `self_update::Certificate` (`from_pem` /
  `from_der`) plus `add_root_certificate` on the `Update`/`ReleaseList` builders and on `Download`
  trust a private/internal CA. A malformed certificate surfaces as `Error::InvalidCertificate`
  from `build()` / `download_to`.
- **`AsyncReleaseUpdate`:** a public sealed trait mirroring `ReleaseUpdate` for the async verbs,
  usable as a generic bound (re-exported at the crate root under `async`).
- **`version::cmp_versions(a, b)`:** a total-order semver comparison returning `Result<Ordering>`.
- **`ReleaseStatus::version()`:** `Option<&str>`, mirroring `VersionStatus::version` (`None` on the
  up-to-date arm).
- **`Releases::from_releases(..)` / `Releases::from_listing(..)`:** build a `Releases` value in
  your own tests.
- **`ReleaseUpdate` is at the crate root:** `self_update::ReleaseUpdate` (was
  `self_update::update::ReleaseUpdate`).
- **Verifying keys are easier:** use the `self_update::VerifyingKey` alias instead of
  `[u8; zipsign_api::PUBLIC_KEY_LENGTH]`, and the `zipsign_api` crate is re-exported as
  `self_update::zipsign_api` (both under the `signatures` feature). The `verifying_keys(...)`
  setter and the `verifying_keys()` accessor now speak this alias directly.
- **The S3 credential type is nameable:** `self_update::backends::s3::AccessKey` (under `s3-auth`)
  is now public. You normally just pass an `(id, secret)` tuple to `.access_key(...)`; the type
  is `#[non_exhaustive]` so a future field (e.g. an STS session token) can be added without a
  break, so build it via the tuple `From` impls, not a struct literal.
- **The S3 credential constructor.** `self_update::backends::s3::AccessKey::new(id, secret)`
  builds a credential as a named alternative to the `(id, secret)` tuple `From` impls (under
  `s3-auth`).
- **Read the installed release without a `match`.** `ReleaseStatus` (renamed from `UpdateStatus`)
  gained `updated_release() -> Option<&Release>` and `into_updated_release() -> Option<Release>`, so
  you can pull the installed `Release` out of `update_extended()`'s result without a `match` (which
  `#[non_exhaustive]` would otherwise force a wildcard arm onto).
- **One call for unattended runs.** Every backend `Update`/`ReleaseList` builder has `.unattended()`,
  which sets `no_confirm(true)` and `show_output(false)` together. The default
  (`no_confirm == false`) blocks on stdin for the interactive confirmation, so a daemon or CI job
  wants `.unattended()` (or the two setters individually).
- **`ArchiveKind` implements `Display`** (rendering `tar.gz` / `zip` / `tar` / `gz` / `plain`),
  used in the `Error::NoSignatures` message in place of the old `Debug` form.
- **Check for an update without installing.** `updater.is_update_available()?` returns
  `Option<Release>`: the newest strictly-newer release, or `None` when up to date, without
  downloading or installing anything (see section 7).
- **Each backend's `ReleaseList` has a documented path:**
  `self_update::backends::<github|gitlab|gitea|s3>::ReleaseList`. The four are distinct types, so
  they are surfaced per backend rather than unified under one crate-root name. Each has
  `fetch_async` under the `async` feature.
- **A custom-backend example** lives at `examples/custom.rs`: a minimal `ReleaseSource` impl with a
  sync update, plus an async variant under the `async` feature.
