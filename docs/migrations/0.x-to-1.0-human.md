# Migrating to `self_update` 1.0

`self_update` 1.0 is the first stable release. It makes a focused set of breaking changes to
clean up the public API before committing to long-term `1.x` stability — this was the only
window in which they could land. Most upgrades are small and mechanical; the sections below
walk through each change, why it was made, and exactly what to do.

> Looking for a terse, copy-paste / automation-friendly checklist? See
> [`0.x-to-1.0.md`](0.x-to-1.0.md).

---

## 1. Builder method names are unified

We removed the `with_` prefix and made the four backends speak the same vocabulary, so you no
longer have to remember which backend calls a thing what.

| Before | After | Where |
|--------|-------|-------|
| `.with_url("https://…")` | `.url("https://…")` | github (its API endpoint) |
| `.with_host("https://…")` | `.url("https://…")` | gitlab, gitea (the instance base URL) |
| `.with_target("x86_64-…")` | `.filter_target("x86_64-…")` | `ReleaseList` on all backends (the release filter) |
| `.access_key_id((id, secret))` | `.access_key((id, secret))` | s3 (`s3-auth`) |
| `.target_version_tag("v1.2.3")` | `.release_tag("v1.2.3")` | every `Update` builder |
| `.identifier("musl")` | `.asset_identifier("musl")` | every `Update` builder |
| `.verifying_checksum(c)` | `.checksum(c)` | every `Update` builder (`checksums`) |
| `.set_progress_callback(cb)` | `.progress_callback(cb)` | `Download` + every `Update` builder |
| `.set_progress_style(t, c)` | `.progress_style(t, c)` | `Download` + every `Update` builder |
| `.set_timeout(d)` | `.timeout(d)` | `Download` |
| `.set_header(n, v)` | `.header(n, v)` | `Download` (single header; `replace_headers` still replaces the whole map) |
| `.show_progress(b)` | `.show_download_progress(b)` | `Download` |
| `.set_headers(map)` | `.replace_headers(map)` | `Download` |

**Why:** the names were tidied so that setters and accessors match (e.g. the `release_tag` /
`asset_identifier` setter now reads back under the same name), so the standalone `Download`
utility's setters line up with the `Update` builder's (`timeout`, `progress_callback`,
`progress_style`, `show_download_progress`), and so every git backend names its endpoint setter
`url(...)` (github's API endpoint, and gitlab/gitea's instance base URL), with the `ReleaseList`
release filter named `filter_target` (distinct from the build-target `target` on the `Update`
builder). These were pure papercuts that a 1.0 is the right time to fix. The old names survive as
`#[doc(alias)]`s, so rustdoc search still finds them.

While here, `request_header` now accepts plain strings as well as typed header values:
`.request_header("X-Foo", "bar")` works, and an invalid header is reported as an `Error::Config`
from `build()` instead of panicking at the call site. The standalone `Download::header` setter
likewise now takes string literals and returns a `Result`:
`download.header("Accept", "application/octet-stream")?`.

**What to do:** find-and-replace the calls. The old names are gone, so the compiler points you
at every site (`no method named with_url …`).

```rust
// before
let releases = self_update::backends::gitlab::ReleaseList::configure()
    .with_host("https://gitlab.example.com")
    .repo_owner("me").repo_name("app")
    .with_target("x86_64-unknown-linux-gnu")
    .build()?;

// after
let releases = self_update::backends::gitlab::ReleaseList::configure()
    .url("https://gitlab.example.com")
    .repo_owner("me").repo_name("app")
    .filter_target("x86_64-unknown-linux-gnu")
    .build()?;
```

## 2. The `Error` enum changed shape

`self_update::errors::Error` is now `#[non_exhaustive]`, and the feature-specific variants were
replaced by two opaque ones:

- `Error::Reqwest(..)` and `Error::Ureq(..)` → **`Error::Http(Box<dyn Error + Send + Sync>)`**
- `Error::StdTimeError`, `Error::TimeError`, `Error::Digest`, `Error::UrlParse` (all the
  `s3-auth` signing errors) → **`Error::S3Auth(Box<dyn Error + Send + Sync>)`**

**Why:** previously an exhaustive `match` on `Error` only compiled for one HTTP-client feature
set — code written for the default `reqwest` build broke if a user switched to `ureq`, and vice
versa, because the variants were `#[cfg]`-gated. Collapsing them into opaque variants makes your
error handling independent of which client/features are enabled. Marking the enum
`#[non_exhaustive]` means we can add error categories in future `1.x` releases without it being
a breaking change.

**What to do:**

```rust
// before
match err {
    Error::Reqwest(e) => log::error!("http: {e}"),
    Error::Release(s) => log::error!("release: {s}"),
    // …one arm per variant…
}

// after
match err {
    Error::Http(e) => log::error!("http: {e}"),
    Error::Release(s) => log::error!("release: {s}"),
    _ => log::error!("other: {err}"), // required: the enum is now non_exhaustive
}
```

Four more variants became opaque the same way: `Error::Zip`, `Error::Signature` (the latter under
the `signatures` feature), `Error::Json`, and `Error::SemVer` now wrap `Box<dyn Error + Send + Sync>`
instead of the concrete `zip::result::ZipError` / `zipsign_api::ZipsignError` / `serde_json::Error` /
`semver::Error`. The arm patterns still match, but you can no longer call an inner-error method on the
bound value directly; inspect it via `source()` or downcast (same as below). And the signatures-only
`Error::NonUTF8` variant is renamed to `Error::SignatureNonUTF8` (rename the arm).

If you need the concrete underlying error, downcast the box or use the standard `source()`:

```rust
if let Error::Http(e) = &err {
    if let Some(re) = e.downcast_ref::<reqwest::Error>() { /* … */ }
}
```

> **Note:** `Error::S3Auth` only exists when you enable the `s3-auth` feature; `Error::Http`
> is always present. If you don't use `s3-auth`, just leave out the `S3Auth` arm — the
> wildcard `_ =>` arm covers it. (Adding an `Error::S3Auth` arm without the feature won't
> compile.)

> **Note:** In 1.0, `Status`, `ArchiveKind`, and `Compression` also became `#[non_exhaustive]`.
> If you have an exhaustive `match` on any of them (especially `Status`, which is returned by
> every `update()` call), add a `_ =>` arm — the compiler will flag any exhaustive match that
> no longer covers all variants. The `Download`, `Extract`, `Move`, and `MoveAll` utility structs
> are also `#[non_exhaustive]` now (their fields were already private, so this only blocks a future
> struct-literal construction), as are each backend's concrete `Update` struct and
> `custom::AsyncUpdate` (the values `build_async()` returns).

> **Note:** the `Status`/`UpdateStatus` predicates were renamed for a consistent pair:
> `uptodate()` to `is_up_to_date()` and `updated()` to `is_updated()`. Rename the calls; the old
> names no longer exist, so the compiler points at each one.

## 3. `ReleaseUpdate` accessors return borrows

The accessor methods on the `ReleaseUpdate` trait now return `&str` / `Option<&str>` instead of
owned `String` / `Option<String>` (`current_version`, `target`, `bin_name`,
`bin_path_in_archive`, `progress_template`, `progress_chars`, `release_tag`, `asset_identifier`,
`auth_token`). `bin_install_path` returns `&Path` instead of an owned `PathBuf`. The
`verifying_keys` accessor (`signatures` feature) similarly returns `&[VerifyingKey]` instead of
`Vec<VerifyingKey>`.

**Why:** these are simple field reads; returning owned `String`s forced an allocation on every
call for no reason. This is the kind of signature that can't be fixed after 1.0, so it had to
happen now.

The `target_version` accessor was also **renamed to `release_tag`** and `identifier` to
`asset_identifier`, each matching its builder setter exactly (so set and get share a name). Same
window-of-opportunity reasoning: a name can't change after 1.0. The standalone `Download`
utility's setters were likewise lined up with the `Update` builder: `set_headers` →
`replace_headers` (it replaces the whole header map), `show_progress` → `show_download_progress`,
`set_timeout` → `timeout`, `set_header` → `header`, `set_progress_callback` → `progress_callback`,
and `set_progress_style` → `progress_style`.

**What to do:** most call sites (`==`, `format!`, passing a `&str`) need no change. Where you
specifically need ownership, add `.to_string()`:

```rust
let v: String = updater.current_version().to_string(); // was: updater.current_version()
```

## 4. `ReleaseUpdate` is now a sealed trait

You can still *call* `ReleaseUpdate` methods — every backend's `build()` still returns a
`Box<dyn ReleaseUpdate>` — but you can no longer `impl ReleaseUpdate for YourOwnType`.

**Why:** the trait was always an internal plumbing abstraction (14 methods, mostly accessors),
never meant as an extension point. Sealing it lets us evolve it in future `1.x` releases without
breaking anyone. (It's also why the `&str` change in §3 was safe to make.)

**What to do:** almost nobody implements this trait. If you did (a custom backend), you'll need
to rework it onto a built-in backend, or open an issue describing the use case — please do, so we
can consider a supported extension mechanism.

## 5. Feature-flag combinations are now enforced

Invalid feature combinations are now a clear compile error instead of a confusing one (or
silent misbehaviour):

- enabling **both** `reqwest` and `ureq` — pick exactly one HTTP client;
- enabling **neither** `reqwest` nor `ureq` (`no HTTP client selected`) — this happens when you
  set `default-features = false` but forget to add a client back; enable exactly one;
- enabling **both** `default-tls` and `rustls` — pick exactly one TLS backend.

**Why:** these were always unsupported; previously "both clients" produced a cryptic symbol
collision and "both TLS" silently picked one. Now you get a one-line message telling you what's
wrong.

**What to do:** if your `Cargo.toml` selects `ureq` or `rustls`, make sure you also set
`default-features = false` (otherwise the default `reqwest` + `default-tls` are still on and
collide):

```toml
# ureq + rustls
self_update = { version = "1.0", default-features = false, features = ["ureq", "rustls"] }
```

## 6. Small removals

- **Two convenience re-exports were dropped:** `self_update::self_replace` and
  `self_update::TempDir`. They pinned downstream code to the crate's exact `tempfile` /
  `self-replace` versions for no real benefit. If you used them, add `tempfile` and/or
  `self-replace` to your own `Cargo.toml` and import them directly (`tempfile::TempDir`,
  `self_replace::self_replace`). The genuinely useful re-exports — `http`, the selected client
  (`reqwest` / `ureq`), and `zipsign_api` — are unchanged.
- `self_update::GetArchiveReaderResult` is no longer public — it was the return type of a
  private function and had no usable purpose. Delete any reference (there almost certainly
  isn't one).
- The deprecated `Error::description()` method is gone (it had been deprecated in std since
  Rust 1.27). Use `Display` (`{err}`) or `err.source()`.
- `should_update(current, latest)` is gone (deprecated since 0.4.2). It was a thin
  version comparison — use `self_update::version::bump_is_greater(current, latest)` for the
  same "is `latest` newer than `current`" check, or `version::bump_is_compatible(..)` if you
  want a semver-compatibility check.
- **s3 `.auth_token(..)` is removed.** The setter is gone from the s3 `Update` builder, so a leftover
  call is now a compile error. The s3 backend signs requests with an `access_key` (AWS SigV4); it
  never used a bearer token, so `auth_token` was always a silent no-op there. Remove the call, and if
  you meant to authenticate to a private bucket, use `.access_key((id, secret))` (the `s3-auth`
  feature) instead. This only affects the s3 backend; `auth_token` still works on
  github/gitlab/gitea.

## 7. Checking for an update returns a `Releases` value

The release-listing methods on the updater now hand back a `Releases` value instead of a bare
`Release` / `Vec<Release>`, and the standalone `is_update_available()` check is gone:

- `get_latest_release()` returns `Releases` (was `Release`).
- `get_latest_releases()` returns `Releases` and no longer takes a `current_version` argument
  (it reads the version you configured). It was `get_latest_releases(current_version)`.
- `get_release_version(tag)` (specific-tag lookup) is unchanged.
- The per-updater `is_update_available()` (and `is_update_available_async()` under `async`) are
  removed. The check now lives on `Releases`.

`Releases` (re-exported as `self_update::Releases`) holds the fetched releases plus your current
version. It has `all() -> &[Release]`, `latest() -> Option<&Release>` (newest first),
`into_vec() -> Vec<Release>`, and `is_update_available() -> Result<bool>`. It is a different type
from the per-backend `ReleaseList` builder.

**Why:** the old `is_update_available()` fetched the release list, then `update()` fetched it again.
Returning the list lets a pre-check reuse a single fetch.

**What to do:** a light "is there a newer release?" pre-check is now one fetch and one call:

```rust
let updater = self_update::backends::github::Update::configure()
    .repo_owner("me").repo_name("app").bin_name("app")
    .current_version(cargo_crate_version!())
    .build()?;

if updater.get_latest_releases()?.is_update_available()? {
    updater.update()?;
}
```

Under the `async` feature the updater has `get_latest_release_async()` /
`get_latest_releases_async()` (returning `Releases`) and `get_release_version_async()` (returning
`Release`); the async pre-check is
`updater.get_latest_releases_async().await?.is_update_available()?`. Where you previously read the
newest release from `get_latest_release()`, call `.latest()` on the returned `Releases`; where you
took the whole list from `get_latest_releases(..)`, call `.all()` or `.into_vec()`.

## 8. Custom backends: a clean sync/async split

If you write a **custom backend** (you `impl ReleaseSource`), two things changed. Most people who
only use the built-in github/gitlab/gitea/s3 backends can skip this section.

**`ReleaseSource` is sync again.** Earlier 1.0 pre-releases bolted three defaulted async helper
methods (`get_latest_release_async`, …) onto `ReleaseSource`. Those are **gone**. `ReleaseSource`
is back to three plain sync fetch methods (and no `Clone` requirement). Instead, the sync and async
worlds are now mirror images of each other, just like the built-in backends' `build()` vs
`build_async()`:

- **Native async source:** implement the new `AsyncReleaseSource` trait — the same three fetches,
  with clean names (`get_latest_release`, `get_latest_releases`, `get_release_version`, **no**
  `_async` suffix). Drive it with the new async custom updater:

  ```rust,ignore
  use self_update::backends::custom::AsyncUpdate;
  let status = AsyncUpdate::configure()
      .source(MyAsyncHost)
      .bin_name("app")
      .current_version("1.0.0")
      .build_async()?
      .update_async().await?;
  ```

  There's no `async-trait` involved — the updater is generic over your source, so the futures cost
  nothing extra. The trait now requires its returned futures to be `Send` (the methods return
  `impl Future<Output = ...> + Send`), so a non-`Send` impl fails to compile at the impl site instead
  of later at the spawn site. A plain `async fn` body satisfies this unless it holds a non-`Send`
  value across an `.await`.

- **Reuse a sync source from async code:** if you already have a `Clone` sync `ReleaseSource` and
  just want to call it from a tokio app, wrap it in `Blocking`:

  ```rust,ignore
  use self_update::backends::custom::{AsyncUpdate, Blocking};
  AsyncUpdate::configure().source(Blocking::new(my_sync_source)) /* ... */;
  ```

  `Blocking` runs your sync fetches on `tokio::task::spawn_blocking` — exactly what the removed
  defaults did, now opt-in and explicit. Its inner field is private: construct it with
  `Blocking::new(source)` and read the wrapped source back with `into_inner()` / `as_inner()`.

**Accessors moved to a `UpdateConfig` supertrait.** The many getter methods on `ReleaseUpdate`
(`bin_name`, `current_version`, `target`, …) now live on a new supertrait
`self_update::UpdateConfig` (and `ReleaseUpdate: UpdateConfig`). Calling them on the usual
`Box<dyn ReleaseUpdate>` value needs **no change**. The only code that breaks is a generic helper
written `fn f<R: ReleaseUpdate>(u: &R)` that calls an accessor — add `use self_update::UpdateConfig;`
and it resolves again.

## Niceties you can now adopt (optional, non-breaking)

These aren't required, but 1.0 makes a few things pleasanter:

- **`http` is re-exported.** Header constants for `Download::header` are reachable as
  `self_update::http::header::ACCEPT` — no need to add the `http` crate yourself, and it works
  the same under `reqwest` or `ureq`.
- **`ReleaseUpdate` is at the crate root:** `self_update::ReleaseUpdate` (was
  `self_update::update::ReleaseUpdate`).
- **Verifying keys are easier:** use the `self_update::VerifyingKey` alias instead of
  `[u8; zipsign_api::PUBLIC_KEY_LENGTH]`, and the `zipsign_api` crate is re-exported as
  `self_update::zipsign_api` (both under the `signatures` feature). The `verifying_keys(...)`
  setter and accessor now speak this alias directly.
- **The S3 credential type is nameable:** `self_update::backends::s3::AccessKey` (under `s3-auth`)
  is now public. You normally just pass an `(id, secret)` tuple to `.access_key(...)`; the type
  is `#[non_exhaustive]` so a future field (e.g. an STS session token) can be added without a
  break — build it via the tuple `From` impls, not a struct literal.
- **`features = ["s3"]` now resolves.** The S3 backend needs no feature (it is always compiled),
  but a no-op `s3` alias feature exists so requesting it doesn't error, matching the other
  backends. Private-bucket signing still needs the real `s3-auth` feature.
- **Check for an update without installing.** Fetch the release list once and ask it:
  `updater.get_latest_releases()?.is_update_available()?` returns a `bool` for whether a newer
  compatible release exists, without downloading or installing anything (see section 7).
- **Each backend's `ReleaseList` has a documented path:**
  `self_update::backends::<github|gitlab|gitea|s3>::ReleaseList`. The four are distinct types, so
  they are surfaced per backend rather than unified under one crate-root name.
- **A custom-backend example** lives at `examples/custom.rs`: a minimal `ReleaseSource` impl with a
  sync update, plus an async variant under the `async` feature.

## Verifying your upgrade

Build and test with your real feature selection:

```bash
cargo build
cargo test
```

If it compiles, you're done — the breaking changes are all surfaced at compile time. The full
list of changes is in [`CHANGELOG.md`](CHANGELOG.md) under `[1.0.0]`.
